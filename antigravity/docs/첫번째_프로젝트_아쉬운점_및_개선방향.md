# 첫 번째 프로젝트(tripartite_section_check.py) 아쉬운 점 및 개선 방향

첫 번째 프로젝트는 같은 영상을 약 50초에 처리할 정도로 이미 빠르지만,  
두 번째 프로젝트(tri_split_detector.py)를 단계별로 더 빠르게 만들기 위해  
**객관적으로 아쉬운 점**과 **어떤 개선을 하면 속도를 더 낮출 수 있는지** 정리한 문서입니다.

---

## 1. 코스 스캔 단계: 프레임당 연산량

### 아쉬운 점

- **매 샘플마다 “정렬 탐색(align tolerance)”을 수행**  
`ALIGN_TOLERANCE_PX`(기본 5)가 0이 아니면, **w/3 ± N 픽셀** 범위에서  
여러 개의 `L`(좌측 너비)에 대해 **매번 crop → split → MSE**를 반복합니다.  
(예: 11개 L 값 × 매 프레임)
- 즉, **한 번 삼분할인지 판별하는 데도 “오프셋 후보 여러 개”를 전부 계산**하고 있어,  
코스 스캔 샘플 수가 많을수록 **불필요한 연산 비율**이 커집니다.
- 가이드(tri_split_guide.md)의 **Phase 2**처럼  
“첫 삼분할에서 한 번만 오프셋을 찾고, 이후에는 **고정 좌표로 absdiff만** 쓰는” 구조가 없습니다.

### 개선 방향

- **Phase 2 도입:**  
코스 스캔에서 **첫 번째로 삼분할이 나온 시점**에서만  
정렬 탐색(또는 template matching)으로 `left_width`(또는 left_x_end, right_x_start)를 구하고,  
**그 다음 샘플부터는 그 좌표를 고정**해서 **crop → absdiff 평균 한 번**만 계산.  
→ 코스 스캔 구간의 대부분에서 “L 후보 여러 개 × MSE” 대신 **단일 absdiff**로 줄일 수 있어 속도 향상.
- **코스 전용 “경량 판정”:**  
코스 스캔용으로는 **고정 1/3·2/3**(align_tolerance=0) 또는 **해상도 더 축소**만 쓰고,  
정렬 탐색은 **경계 정밀화 단계**나 **첫 캘리브레이션**에서만 수행하는 식으로 단계 분리.

---

## 2. 병렬화와 디스크 병목

### 아쉬운 점

- **기본 워커 수가 `os.cpu_count()`**  
코어 수만큼 프로세스를 띄우면 **같은 영상 파일**을 여러 FFmpeg가 동시에 서로 다른 시점으로 읽습니다.  
**7200rpm HDD**처럼 랜덤 읽기 성능이 낮은 저장장치에서는  
디스크가 병목이 되어 **병렬화 이득이 거의 없거나 오히려 느려질 수** 있습니다.
- **저장소 타입(HDD/SSD)에 따른 자동 조정이 없음**  
사용자가 `--workers 1`을 직접 주지 않으면, HDD에서도 코어 수만큼 워커가 돌아갈 수 있어  
“같은 50초”를 목표로 해도 환경에 따라 **체감 속도가 들쭉날쭉**할 수 있습니다.

### 개선 방향

- **워커 수 기본값을 보수적으로** 두거나,  
**환경 변수/설정**으로 “HDD 모드”(예: workers=1)를 선택할 수 있게 하기.  
또는 (구현 난이도는 높지만) **디스크 랜덤 읽기 성능을 간단히 측정**해  
낮으면 워커를 1~2로 제한하는 휴리스틱 도입.
- **두 번째 프로젝트**에서는  
“일단 순차(workers=1)로 설계하고, SSD/NVMe에서만 병렬 옵션을 켜는” 식으로  
디스크 병목을 피하는 전략을 선택할 수 있습니다.

## 3. 캐시 설계

### 아쉬운 점

- `**--coarse-cache` 시 프레임을 디스크(.npy)에 저장**  
코스 스캔 결과를 경계 정밀화에서 재사용할 때,  
**같은 영상에서 같은 시점을 다시 읽는 대신** 디스크에 써 둔 `.npy`를 읽습니다.  
다만 **캐시 자체가 디스크 I/O**이기 때문에,  
HDD에서는 **캐시 쓰기/읽기 비용**이 꽤 커질 수 있습니다.
- **캐시가 “원본 해상도”일 때만 경계에서 재사용**  
`--coarse-scale`을 쓰면 코스는 저해상도로 돌리지만,  
경계 정밀화는 원본 해상도로 다시 추출하므로,  
**캐시를 켜도 코스에서 쓴 저해상도 프레임은 경계에서 쓰이지 않음**.  
“코스만 저해상도 + 캐시 없음”이면 경계에서 **전부 재추출**이라,  
코스에서 이미 본 시점을 한 번 더 FFmpeg로 부르는 비용이 그대로 남습니다.

### 개선 방향

- **경계 정밀화 시 “코스 시점 근처”만 재추출**하는 현재 구조는 유지하되,  
캐시를 쓸 때 **저해상도 캐시는 쓰지 않고**,  
**원본 해상도 캐시만 선택적으로** 쓰거나(예: NVMe/SSD일 때만 캐시 켜기),  
또는 **메모리 캐시**(프로세스 내 dict/배열)로만 재사용하고 디스크 캐시는 제거하는 옵션을 두기.
- **두 번째 프로젝트**에서는  
“캐시 없이, 코스 구간 리스트만 넘기고 경계에서만 필요한 시점을 최소 횟수로 추출”하는 식으로  
I/O를 단순화할 수 있습니다.

## 4. 판정 방식: MSE vs Template Matching

### 아쉬운 점

- **1/3·2/3 고정 분할 + MSE**  
“좌·우가 같은지”를 **고정된 1/3, 2/3** 또는 **L 후보 여러 개**에 대해 MSE로만 판별합니다.  
실제 송출에서는 **분할선이 640/1280이 아니라 637/1273**처럼 **몇 픽셀 어긋난 경우**가 있는데,  
이때 **“어디가 실제 경계인지”를 한 번에 찾아주는** template matching 같은 방식을 쓰지 않습니다.
- **정렬을 “L 후보 열거”로만 처리**  
`align_tolerance_px`로 L을 여러 개 시도하는 방식은  
**정확한 오프셋 한 개를 빠르게 찾는** template matching보다  
**프레임당 연산 횟수**가 많을 수 있습니다.  
(특히 코스 스캔에서 매번 이걸 하면 비용이 누적됨.)

### 개선 방향

- **두 번째 프로젝트**처럼  
**“좌측 영역을 템플릿으로 해서 우측에서 매칭”**하는 방식으로  
**한 번에 최적 오프셋**을 구하고,  
그 다음에는 **고정 좌표 + absdiff**만 쓰는 흐름이  
“정확도 + 프레임당 연산 최소화” 측면에서 유리할 수 있습니다.
- 첫 프로젝트를 유지한다면:  
**코스 스캔에서는 align_tolerance=0(고정 1/3)**으로 두고,  
**경계 정밀화나 “첫 캘리브레이션”**에서만 L 탐색을 쓰는 식으로 역할을 나누면  
코스 구간의 연산량을 줄일 수 있습니다.

## 5. 키프레임·병합 단계

### 아쉬운 점

- **키프레임 목록을 “잘린 조각 파일” 기준으로 조회**  
병합 시 각 구간을 잘라낸 **임시 파일**에 대해  
`_get_keyframe_times_from_file()`을 호출해 키프레임 시각을 가져옵니다.  
**긴 원본**에서 키프레임만 한 번에 뽑는 방식이 아니라,  
**구간 수만큼** 잘린 파일이 생기고, 그만큼 ffprobe(패킷/프레임 열거)가 반복될 수 있어  
**구간이 많을수록** 키프레임 조회 비용이 늘어납니다.
- **가이드(tri_split_guide.md) §5의 “키프레임 스냅”**  
“검출된 start/end를 **가장 가까운 I-Frame으로 확장**”하는 로직이  
첫 프로젝트에는 **병합 단계 안에만** 녹아 있고,  
**전체 파이프라인에서 “키프레임 목록 1회 확보 → 구간별 스냅”**이  
문서처럼 단계적으로 드러나 있지 않을 수 있습니다.  
(구현은 되어 있으나, 원본 전체 키프레임을 한 번만 쓰는 구조인지는 코드 경로에 따라 다름.)

### 개선 방향

- **원본 파일 하나**에 대해 **키프레임 타임스탬프를 1회만** ffprobe 등으로 얻어 두고,  
검출된 (start, end)마다 **그 리스트에서 가장 가까운 I-Frame**을 찾아  
`final_start`, `final_end`로 보정한 뒤,  
**그 시간 기준으로만** 잘라내기/병합에 사용하는 구조가  
가이드와 맞고, ffprobe 호출 횟수도 줄일 수 있습니다.
- 두 번째 프로젝트에서는  
“구간 검출 → 키프레임 리스트 1회 조회 → 구간별 스냅 → 추출·병합” 순서를  
문서와 코드에서 명확히 분리해 두면,  
나중에 “키프레임 단계만” 최적화하기 쉬워집니다.

## 6. 코드 구조·유지보수

### 아쉬운 점

- **한 파일에 1500줄 이상**  
CLI, ffprobe, 프레임 추출, CPU/GPU 판정, 코스 스캔, 경계 정밀화, 병합, 검증, 로그 등이  
한 스크립트에 모여 있어,  
**“코스 스캔만 빠르게 바꾸기”**, **“판정 로직만 MSE → template matching으로 바꾸기”** 같은  
**부분 최적화**가 어렵고, 테스트도 무거워지기 쉽습니다.
- **옵션과 분기**  
`--workers`, `--coarse-scale`, `--coarse-cache`, `--cuda`, `--strict`, `--align-tolerance` 등  
조합이 많아서, **“가장 빠른 경로”**가 어떤 옵션 조합인지 사용자가 파악하기 힘들 수 있습니다.

### 개선 방향

- **두 번째 프로젝트**에서는  
**단계를 나눈 문서(진행상태_및_할일.md)**에 맞춰  
**“전체 스캔(대략) → 시작 정밀화 → 끝 정밀화 → 키프레임 보정 → 추출·병합”**을  
**함수/모듈 단위로 분리**해 두면,  
각 단계만 따로 최적화·교체하기 쉽습니다.
- **“기본은 가장 빠른 경로”**  
예: HDD에서도 무난한 **순차 + rawvideo 파이프 + Phase 2(오프셋 Lock)** 를 기본으로 두고,  
**SSD/NVMe·다중 코어**에서만 `--workers`, `--coarse-scale` 등을 켜는 식으로  
**옵션 수를 줄이거나 기본 동작을 단순화**할 수 있습니다.

---

## 7. 요약: 무엇을 개선하면 더 빨라질 수 있는가


| 구분           | 아쉬운 점                                  | 개선 방향 (속도 관점)                                  |
| ------------ | -------------------------------------- | ---------------------------------------------- |
| **코스 스캔 연산** | 매 샘플마다 정렬 탐색(L 후보 × MSE)               | Phase 2: 첫 구간에서만 오프셋 확보 후, 고정 좌표 + absdiff만 사용 |
| **병렬화**      | 기본이 cpu_count(), HDD에서 디스크 병목          | HDD 시 워커 1 또는 보수적 기본값; SSD에서만 병렬 강화            |
| **캐시**       | 코스 캐시가 디스크 .npy → HDD에서 I/O 부담         | 메모리 캐시만 쓰거나, 캐시 사용을 SSD/NVMe로 제한               |
| **판정**       | 고정 1/3 + L 후보 열거, template matching 없음 | 템플릿 매칭으로 오프셋 1회 계산 → 이후 고정 좌표 비교               |
| **키프레임**     | 구간별로 잘린 파일에 ffprobe 반복 가능성             | 원본 1회 키프레임 목록 → 구간별 스냅만 적용                     |
| **구조**       | 단일 대형 스크립트, 단계 혼재                      | 단계별 모듈/함수 분리, “가장 빠른 경로” 기본화                   |


이 문서는 **첫 번째 프로젝트를 비난하기 위한 것이 아니라**,  
**두 번째 프로젝트에서 “어디를 더 개선하면 속도와 유지보수성을 더 끌어올릴 수 있는지”**  
객관적으로 정리한 참고용입니다.  
첫 번째 프로젝트의 **장점**(rawvideo 파이프, 병렬화, 단계별 타이밍 등)은  
`tripartite_vs_detector_성능비교.md`에 이미 정리되어 있습니다.
