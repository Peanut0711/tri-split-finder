# 삼분할 구간 검출·병합 파이프라인 정리

## 1. 전체 시퀀스 요약

사용자가 수행한 두 단계는 다음과 같다.


| 단계                   | 명령                                                                                                                | 목적                                                  |
| -------------------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| **1) 분석 + 구간 목록 저장** | `python tripartite_section_check.py "\\...\dbsek28_20260305_194432.ts" --cuda --workers 16 --segments-out`        | 전체 영상에서 삼분할 구간 검출 후 8개 구간을 `*_seg.txt`에 저장          |
| **2) 목록 편집 후 병합**    | `python tripartite_section_check.py "\\...\dbsek28_20260305_194432.ts" --cuda --workers 16 --segments-in --merge` | `*_seg.txt`에서 사용자가 남긴 2개 구간만 읽어, 코덱 카피로 잘라 한 파일로 병합 |


결과: 3시간 11분 영상 → 검출 8개 구간 → 사용자가 2개만 선택 → 약 10분 분량의 `*_merged.mp4` 생성.

## 2. 파이프라인 구조

### 2.1 모드 A: 검출 실행 (--segments-out만, 또는 --merge까지)

```
[입력 영상]
     │
     ▼
┌─────────────────┐
│ 1. 메타(ffprobe) │  ← 길이, 해상도
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 시점 목록     │  ← coarse_interval(30s)마다 타임스탬프 생성 → 384개
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 코스 스캔     │  ← 각 시점에서 1프레임 추출 → 삼분할 여부 판정 (병렬 16프로세스)
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. 코스 정렬     │  ← 시점 순 정렬 (병렬 시에만)
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. 후보 구간 수집 │  ← 연속 True 구간을 (t_start_cand, t_end_cand)로 묶음
└────────┬────────┘
         ▼
┌─────────────────┐
│ 6. 경계 정밀화   │  ← 후보별로 시작/끝을 이진 탐색(0.5초 정밀도) → 정확한 (start,end)
└────────┬────────┘
         ▼
┌─────────────────┐
│ 7. 병합(인접)    │  ← 10초 이내 인접 구간 하나로 묶음
└────────┬────────┘
         ▼
┌─────────────────┐
│ 8. 최소 길이 필터 │  ← 20초 미만 구간 제거
└────────┬────────┘
         ▼
[구간 목록] ──(선택)──► --segments-out → *_seg.txt
                    ──(선택)──► --merge → ffmpeg 병합
```

### 2.2 모드 B: 구간 목록 읽기 + 병합 (--segments-in [--merge])

```
[입력 영상]  [*_seg.txt (사용자 편집)]
     │              │
     ▼              ▼
┌─────────────────────┐
│ load_segments_from_file │  ← 검출 단계 전부 생략
└──────────┬──────────┘
           ▼
     [구간 목록 2개]
           │
           ▼ (--merge 시)
┌─────────────────────────────────────────────────────────┐
│ merge_segments 파이프라인                                 │
│  for each 구간:                                          │
│    ① ffmpeg -ss start_cut -t duration_cut -c copy → temp │
│    ② ffprobe 키프레임 목록 → 첫 키프레임까지 잘라 segment  │
│  concat demuxer로 segment_0001.ts ... 이어붙임 → merged  │
└─────────────────────────────────────────────────────────┘
           ▼
     [*_merged.mp4]
```

정리하면, **검출 파이프라인**과 **병합 파이프라인**이 나뉘며, `--segments-in`을 쓰면 검출 없이 “구간 목록 읽기 → (선택) 병합”만 수행한다.

## 3. 소요 시간이 오래 걸리는 이유

### 3.1 검출 단계 (예: 총 약 2분 2초)


| 단계          | 소요(예시)     | 비율        | 이유                                                                                                 |
| ----------- | ---------- | --------- | -------------------------------------------------------------------------------------------------- |
| 메타(ffprobe) | 0.19초      | 0.2%      | 한 번의 메타 조회라 무시 가능                                                                                  |
| 시점 목록       | ~0초        | -         | 메모리 상 계산만                                                                                          |
| **코스 스캔**   | **1분 46초** | **87.4%** | **384개 시점 × (ffmpeg 1프레임 추출 + 삼분할 판정)**. 시점 수 = 영상길이/30초. 각 시점마다 디코딩·디코더 이동이 필요해 병렬해도 I/O·디코딩이 병목. |
| 코스 정렬       | ~0초        | -         | 리스트 정렬만                                                                                            |
| 후보 구간 수집    | ~0초        | -         | 리스트 한 번 순회                                                                                         |
| **경계 정밀화**  | **15.26초** | **12.5%** | **후보 8개 × (시작 이진탐색 + 끝 이진탐색)**. 탐색마다 `extract_frame` 호출. 0.5초 정밀도면 구간당 수~十数次 호출.                   |
| 병합·최소길이 필터  | ~0초        | -         | 리스트 처리만                                                                                            |


- **코스 스캔**: 영상 길이에 비례해 시점 수가 늘고, 각 시점마다 **ffmpeg로 1프레임 추출**이 필수라, 네트워크 경로(`\\192.168.8.113\...`)면 디코딩 + I/O가 쌓여 시간이 길어짐.
- **경계 정밀화**: 구간마다 **이진 탐색**으로 정확한 시작/끝을 찾기 위해 추가로 여러 번 프레임 추출이 일어남.

### 3.2 병합 단계 (예: 총 약 140초)


| 작업             | 소요(예시) | 이유                                                |
| -------------- | ------ | ------------------------------------------------- |
| 구간 [1] 추출      | 3.3초   | ffmpeg `-c copy`로 잘라 쓰기. 네트워크 디스크 I/O.            |
| 구간 [1] 키프레임 트림 | -      | ffprobe로 키프레임 탐색 + ffmpeg로 재절단                    |
| 구간 [2] 추출      | 43.6초  | 긴 구간(약 574초)을 네트워크에서 읽고 쓰는 시간.                    |
| 구간 [2] 키프레임 트림 | -      | 동일                                                |
| concat 합치기     | 51.0초  | 2개 TS를 concat demuxer로 합쳐 한 MP4로 쓰기. 역시 네트워크 I/O. |


- **입·출력이 네트워크 경로**이므로 디스크 대역폭과 지연이 그대로 반영된다.
- 구간 추출은 **순차 처리**라, 구간이 많을수록 총 시간이 선형으로 늘어난다.
- 각 구간마다 **ffprobe(키프레임) + ffmpeg(트림)** 이 한 번씩 더 돌아간다.

## 3.3 파레토 법칙에 따른 우선 개선 순서

**가장 오래 걸리는 작업부터 손보면, 적은 수의 개선으로 전체 체감 시간을 크게 줄일 수 있다.** 아래는 예시 로그 기준으로 **소요 시간이 긴 순**으로 정렬한 목록이다. 이 순서대로 개선을 진행하면 된다.


| 순위    | 작업           | 예시 소요             | 비고                                         |
| ----- | ------------ | ----------------- | ------------------------------------------ |
| **1** | **코스 스캔**    | **1분 46초 (106초)** | 검출 1회 시 전체의 **87.4%**. 384시점 × 프레임 추출·판정.  |
| **2** | **병합 전체**    | **2분 20초 (140초)** | 병합 1회 실행 시. (검출과 별도 실행이므로, 병합만 쓸 때의 총 시간.) |
| 2-1   | └ concat 합치기 | 51초               | 병합 내에서 가장 긴 단일 작업.                         |
| 2-2   | └ 긴 구간 추출    | 43.6초             | 한 구간(약 574초) 추출. 구간 수·길이에 비례.              |
| 2-3   | └ 짧은 구간 추출   | 3.3초              | 짧은 구간 1개.                                  |
| 2-4   | └ 키프레임 트림 등  | (미측정)             | 구간당 ffprobe + ffmpeg 트림.                   |
| **3** | **경계 정밀화**   | **15.3초**         | 검출 1회 시 전체의 **12.5%**. 후보 8개 × 이진 탐색.      |
| 4     | 메타(ffprobe)  | 0.2초              | 무시 가능.                                     |


**실전 적용 요약**

- **검출 단계를 줄이고 싶을 때** → **1번 코스 스캔**을 가장 먼저 개선 (저해상도 추출, `--coarse-interval` 조정 등).
- **병합 단계를 줄이고 싶을 때** → **2번 병합**에서 **concat(51초)** 와 **긴 구간 추출(43.6초)** 순으로 개선 (로컬 SSD 출력, 구간 병렬 추출 등).
- 그 다음으로 **3번 경계 정밀화**를 다루면, 검출 시간이 추가로 줄어든다.

#### 코스 스캔 최적화 후: 다음에 손볼 구간

코스 스캔을 **저해상도(--coarse-scale 640)** 로 이미 최적화한 상태라면, 검출 파이프라인에서 남는 구간은 아래 순서로 보는 것이 좋다.


| 우선순위  | 구간            | 현재 비중(대략)                        | 추천 방향                                                                                                         |
| ----- | ------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **1** | **경계 정밀화**    | 검출 전체의 **20~30%** (예: 40초 중 10초) | 이진 탐색 반복 상한, 정밀도 완화 옵션(`--boundary-tolerance SEC`). 원본 해상도만 쓸 때는 `--coarse-cache`(캐시를 로컬에)로 경계 ffmpeg 호출 일부 감소. |
| **2** | **병합(merge)** | 병합 1회 실행 시 **전체 시간** (1~2분대)     | 검출과 별도 실행이므로 “검출이 끝난 뒤” 체감 대상. 구간별 병렬 추출, 출력을 로컬 SSD로.                                                        |


- **검출만** 더 줄이고 싶다면 → **경계 정밀화**부터 (이진 탐색 횟수/정밀도 조정, 필요 시 캐시는 원본 해상도+로컬에서만).
- **자르기·병합(--merge)** 이 오래 걸린다면 → **병합 단계** (구간 추출 병렬화, 출력 경로 로컬화).

## 4. 개선 가능 포인트 (꾸준히 최적화할 부분)

### 4.1 코스 스캔

- **저해상도로 프레임 추출**:  
ffmpeg에 `-vf scale=640:-1` 등으로 줄여서 파이프로 넘기고, `extract_frame` 쪽에서 리사이즈된 해상도로 받아서 사용. 디코딩·메모리·판정 연산 모두 감소. 실험용 해상도·예상 속도는 아래 별도 문단 참고.
- **간격 조정**:  
`--coarse-interval 60` 등으로 시점 수를 줄이면 검출 정밀도는 약간 떨어질 수 있지만, 경계 정밀화가 보정하므로 긴 영상에서는 체감 시간이 크게 줄어들 수 있음. (고려 대상이 아니면 생략 가능.)
- **프레임 캐시(실험)** — 아래 별도 설명.

#### 프레임 캐시 개념

지금 구조에서는 **코스 스캔**과 **경계 정밀화**가 같은 영상을 두 번 읽는다.

- **코스 스캔**: 시점 0, 30, 60, … 초마다 `extract_frame`으로 1프레임씩 뽑아서 삼분할 여부만 판정하고, **프레임 자체는 버린다**.
- **경계 정밀화**: 후보 구간마다 이진 탐색으로 정확한 시작/끝을 찾을 때, 22.5초, 26.25초 같은 **코스 시점 사이의 시간**에서 다시 `extract_frame`을 호출한다. 이때마다 ffmpeg가 새로 seek·디코딩을 한다.

**프레임 캐시**는, 코스 스캔 때 뽑은(또는 그 주변) 프레임을 **메모리나 디스크에 저장해 두었다가**, 경계 정밀화에서 “이 시간 근처 프레임이 필요하다”고 할 때 **캐시에서 꺼내 쓰자**는 아이디어다. 그러면 경계 정밀화 구간에서 ffmpeg 호출을 줄일 수 있다.

- **왜 “시점 주변”인가**: 코스는 30초 간격(0, 30, 60, …)만 쓰는데, 경계 정밀화는 그 **사이** 시간(예: 15, 22.5, 26.25)을 요구한다. 그래서 “30초 한 장만 캐시”하면 15초·22초 요청에는 못 쓴다. 그래서 **각 코스 시점 t 주변(예: t−2초 ~ t+2초) 구간을 한 번에 디코딩해 여러 프레임을 캐시**하는 식으로 설계하게 된다.
- **트레이드오프**: (1) 코스 스캔 시 1프레임이 아니라 짧은 구간을 디코딩·저장하므로 코스 스캔 비용과 메모리(또는 디스크) 사용이 늘어남. (2) 경계 정밀화는 `ProcessPoolExecutor`로 별도 프로세스라, 메인에서 만든 캐시를 그대로 공유하기 어렵고, 캐시를 인자로 넘기거나 디스크 캐시를 두는 설계가 필요함. (3) 구현 복잡도가 커진다.

정리하면, **“같은 구간을 경계 정밀화에서 다시 읽지 않게 하려고, 코스 스캔 시점 주변 프레임을 저장해 두고 나중에 재사용하는 것”**이 프레임 캐시 개념이다. 코스 스캔 시간을 줄이는 게 1순위라면, 저해상도 추출을 먼저 적용하고, 프레임 캐시는 실험·선택적으로 검토하면 된다.

**구현된 캐시 (`--coarse-cache`)**  

- 코스 스캔 시 각 시점의 프레임을 **임시 폴더**에 `.npy`로 저장한다. 폴더는 `tripartite_cache_...` 형태로 입력 파일 부모 디렉터리(또는 시스템 임시 디렉터리)에 생성되며, **작업이 끝나면 자동 삭제**된다. (캡처 사진이 아니라 numpy 배열만 저장하므로 용량은 시점 수×해상도에 비례.)
- **경계 단계에서 캐시를 “쓴다” vs “안 쓴다”**  
  - **캐시를 실제로 쓰는 경우** (`--coarse-cache`만, `--coarse-scale` 없음): 코스 스캔이 **원본 해상도**로 프레임을 캐시한다. 경계 정밀화에서 “이 시각의 프레임 필요”할 때, **캐시 시각과 0.5초 이내면 디스크에서 캐시를 읽어 쓴다** → 이때는 ffmpeg 추출(extract)을 하지 않는다. 0.5초보다 멀면 캐시 미스라서 그때만 원본 해상도로 **추출**한다.  
  - **캐시를 안 쓰는 경우** (`--coarse-scale 640 --coarse-cache`): 코스 스캔은 640×360으로 돌리고 그 해상도만 캐시에 저장한다. 경계 정밀화는 **항상 원본 해상도**가 필요하므로, 캐시(640)는 해상도가 달라서 **경계에서는 사용하지 않고**, 경계 단계에서는 **매번 원본 해상도로 ffmpeg 추출**만 한다. 즉 “경계는 원본 해상도로 추출”은 **저해상도+캐시**일 때만 해당하는 말이다.

#### 캐시를 안 쓸 때 vs 쓸 때: 논리와 이점

**캐시를 안 쓸 때 (기본 동작)**  

- **코스 스캔**: 시점 0, 30, 60, … 초마다 ffmpeg로 1프레임씩 뽑아서 삼분할 여부만 판정한 뒤 **프레임은 버린다**.  
- **경계 정밀화**: 후보 구간마다 이진 탐색으로 “정확한 시작/끝”을 찾는다. 이때 필요한 시각은 15초, 22.5초, 26.25초처럼 **코스 시점 사이**인 경우가 많다. 그 시각의 프레임은 코스에서 한 번도 안 뽑았으므로 **없다**. 그래서 경계 단계에서 **그때그때 ffmpeg로 다시 추출**한다.  
- **결과**: 같은 영상을 “코스용”으로 한 번(383회 추출), “경계용”으로 또 한 번(이진 탐색마다 추출) 읽는 셈이라, **경계 정밀화 구간에서 ffmpeg 호출이 많이 발생**한다.

**캐시를 쓸 때 (--coarse-cache, 원본 해상도)**  

- **코스 스캔**: 0, 30, 60, … 초에서 뽑은 프레임을 **판정에 쓴 뒤 디스크에 저장**해 둔다. (코스는 조금 느려질 수 있음: 저장 비용.)  
- **경계 정밀화**: “t초 프레임 필요”할 때, **저장된 시각 중 t와 0.5초 이내가 있으면** 그 파일을 읽어 쓴다 → **ffmpeg 호출 없음**. 0.5초 이내인 게 없으면(캐시 미스) 그때만 ffmpeg로 추출한다.  
- **이점**: 이진 탐색이 **우연히 30초, 60초 근처**(예: 29.8초, 60.2초)를 요청하는 경우가 있으면, 그만큼 **경계 단계의 ffmpeg 호출이 줄어든다**. “같은 영상을 두 번 읽는” 양을 줄이는 효과. 캐시 적중 비율이 높을수록 경계 정밀화 시간이 줄어든다.

**캐시를 쓰지만 저해상도로만 저장할 때 (--coarse-scale 640 --coarse-cache)**  

- **코스 스캔**: 640×360으로 빠르게 뽑고, 그 프레임만 캐시에 저장한다.  
- **경계 정밀화**: 정확한 경계를 찾으려면 **원본 해상도**가 필요하다. 캐시에는 640×360만 있으므로 **경계에서는 캐시를 쓰지 않고**, 항상 원본 해상도로 추출한다.  
- **이점**: 이 경우 캐시는 **경계 단계 시간을 줄이는 데 기여하지 않는다**. 이점은 **코스 스캔을 저해상도로 해서 빨리 돌리는 것**뿐이다. 캐시는 “저장은 하되 경계에서는 사용하지 않는” 상태라, 실질적 이점은 저해상도 옵션에서 온다.

**요약**  

- **캐시 미사용**: 코스에서 쓴 프레임을 버리므로, 경계에서 필요한 시각마다 전부 새로 추출 → 경계 구간 부담이 큼.  
- **캐시 사용(원본 해상도)**: 코스에서 쓴 프레임을 남겨 두고, 경계에서 “가까운 시각”이면 재사용 → 경계 구간 ffmpeg 호출 일부 감소.  
- **캐시 사용(저해상도만)**: 경계는 원본이 필요해서 캐시를 안 씀 → 경계 이점 없음, 코스만 저해상도로 빨라지는 이점만 있음.

#### 저해상도 사용 시: 캐시 켜면 오히려 느려짐 (실측)

동일 영상 `dbsek28_20251224_171035.ts` (03:27:24, 1920×1080, 416 시점)에 대해 **--coarse-scale 640**만 켠 경우와 **--coarse-scale 640 --coarse-cache** 둘 다 켠 경우를 비교한 결과다.


| 항목        | 저해상도 + 캐시 (640 + cache) | 저해상도만 (640) |
| --------- | ----------------------- | ----------- |
| 코스 스캔     | **2분 4초**               | **30.1초**   |
| 경계 정밀화    | 17.6초                   | 10.0초       |
| **전체**    | **2분 22초**              | **40.1초**   |
| 삼분할 구간 결과 | 6개 (동일)                 | 6개 (동일)     |


**분석**  

- 저해상도만 쓸 때 코스 스캔은 **30초**인데, 캐시까지 켜면 **2분 4초**로 약 **4배** 늘어났다.  
- 이유: 캐시를 쓰면 코스 스캔 시 **416개 프레임을 전부 디스크에 .npy로 저장**한다. 이번 실측에서는 캐시 폴더가 **네트워크 경로**(`\\192.168.8.113\ffmpeg\tripartite_cache_...`)에 만들어져, **416번의 네트워크 쓰기**가 발생했다. 네트워크 지연·대역폭 때문에 코스 스캔 시간이 크게 늘어난 것이다.  
- 경계 정밀화에서는 **저해상도 캐시를 사용하지 않으므로** (원본 해상도 필요), 캐시를 켜도 경계 단계가 빨라지지 않는다.  
- 따라서 **저해상도(--coarse-scale)를 쓸 때는 --coarse-cache를 쓰지 않는 것이 훨씬 빠르다.** 캐시는 “경계에서 재사용” 이점이 없는데, 코스 단계에만 “416회 네트워크 쓰기” 부담이 추가되기 때문이다.

**권장**  

- **--coarse-scale 640** (또는 960/1280)만 사용 → **--coarse-cache 생략**.  
- 캐시를 쓰고 싶다면 **원본 해상도**로 코스 스캔할 때만(`--coarse-cache`만, `--coarse-scale` 없이) 사용하고, 캐시 폴더는 **로컬 디스크**에 두는 것이 좋다.

#### 저해상도 추출 실험 (FHD → 어느 정도까지 줄일지)

대상은 **FHD(1920×1080)** 영상이다. 아래는 실험에 쓸 만한 목표 해상도와, 기대할 수 있는 효과를 정리한 것이다.


| 목표 해상도       | 픽셀 수 비율(FHD 대비) | 비고                                                                                                   |
| ------------ | --------------- | ---------------------------------------------------------------------------------------------------- |
| **1280×720** | 약 1/2.25 (44%)  | 720p. 보수적. 삼분할 판정 신뢰도 유지에 유리.                                                                        |
| **960×540**  | 약 1/4 (25%)     | qHD. 속도·품질 균형 후보.                                                                                    |
| **640×360**  | 약 1/9 (11%)     | **권장 실험 기본값.** 파이프·판정 부담 크게 감소.                                                                      |
| 480×270 이하   | 1/16 이하         | 너비가 작아지면 `ALIGN_TOLERANCE_PX`(10px)가 상대적으로 커져, 좌/우 정렬 탐색이 너무 성기게 될 수 있음. 필요 시 픽셀 허용치를 비율로 바꾸면 검토 가능. |


**실험 순서 제안**: 먼저 **640×360** (`-vf scale=640:-1`)으로 적용해 보고, 검출 품질이 괜찮으면 유지, 부족하면 **960×540** 또는 **1280×720**으로 올려서 비교.

**저해상도에서 실제로 줄어드는 부분**

- **ffmpeg → Python 파이프**: 출력 픽셀 수에 비례해 전송량 감소. (예: 640×360이면 FHD 대비 약 1/9.)
- **Python 쪽**: `crop_middle_y` → `split_third` → `mse` 등 픽셀 수에 비례해 연산량 감소.
- **디코딩**: ffmpeg는 보통 1프레임 풀해상도 디코딩 후 `scale` 필터로 줄이므로, **디코딩 비용 자체는 거의 그대로**일 수 있음. 네트워크·디스크 I/O가 크면 전체에서 디코딩이 차지하는 비중이 커져, 저해상도 효과가 제한적일 수 있음.

**예상 속도 향상률 (코스 스캔 구간 기준)**

- **단순 모델**: 프레임당 시간 = (디코딩) + (파이프 전송) + (Python 판정). 저해상도는 **파이프·판정**만 줄이고 디코딩은 유지한다고 가정.
- **디코딩이 대부분(예: 70%)인 경우** (네트워크/느린 디스크 등): 파이프·판정이 30% → 640×360이면 그 30%가 약 1/9 수준으로 줄어들어 전체 약 **73%** 수준 → **약 15~25% 단축**.
- **파이프·판정이 꽤 있는 경우(예: 40~50%)**: 640×360이면 **약 30~40% 단축**까지도 가능.
- **로컬 SSD + CPU 디코딩**처럼 파이프·판정 비중이 더 크면, **40% 전후 단축**도 기대할 수 있음.

정리하면, **실험은 640×360을 기본으로 하고, 960×540·1280×720은 품질 보정용으로 비교**하면 된다. **예상 코스 스캔 단축은 환경에 따라 대략 15~40%** 구간으로 두고 측정해 보는 것이 좋다.

#### 저해상도 코스 스캔 실측 비교 (옵션 유무)

동일 영상 `dbsek28_20260108_221503.ts` (03:10:53, 1920×1080)에 대해 **원본 해상도**와 **--coarse-scale 640**을 각각 실행한 결과 비교.


| 항목                       | 원본 해상도 (옵션 없음) | 저해상도 (--coarse-scale 640) |
| ------------------------ | -------------- | ------------------------- |
| 코스 스캔                    | 1분 33초         | **21.3초**                 |
| 경계 정밀화                   | 15.7초          | 9.3초                      |
| **전체 실행**                | **1분 48초**     | **30.6초**                 |
| 발견 구간 수(경계 정밀화 직후)       | 9개             | 9개                        |
| 최종 삼분할 구간 수(20초 필터·병합 후) | 7개             | 7개                        |


**삼분할 구간 오차 여부**

- **발견(경계 정밀화) 타임스탬프**: 9개 모두 **동일** (00:14:54~~00:15:11, 00:15:34~~00:16:09, 00:24:26~~00:25:15, 00:26:20~~00:26:31, 01:15:56~~01:17:00, 01:29:58~~01:30:57, 01:33:52~~01:34:22, 01:35:34~~01:36:30, 01:38:50~01:39:53).
- **최종 7개 구간**: 시작·끝 시각 및 길이 **완전 일치** (35.6초, 49.7초, 63.8초, 59.5초, 29.5초, 55.8초, 63.8초).

**결론**: 이 실측에서는 **저해상도 옵션 유무에 따른 삼분할 구간 오차가 없음**. 640×360 코스 스캔으로도 동일한 7개 구간이 동일 타임스탬프로 검출되었고, 전체 실행 시간은 **약 1분 48초 → 30초**로 크게 단축되었다 (코스 스캔 약 **78% 단축**, 전체 약 **83% 단축**).

#### 저해상도에서도 “약간 밀린” 삼분할을 잡을 수 있는가 (논리)

현재 코드는 **삼분할 구간이 약간 밀린 경우**를 두 가지로 감안하고 있다.

1. **경계 정렬 탐색 (`ALIGN_TOLERANCE_PX = 10`)**
  송출자가 1/3·2/3가 아니라 647/1273 픽셀처럼 그린 경우를 대비해, **좌측 너비 L**을 `w/3 ± 10` 픽셀 구간에서 바꿔 가며 MSE가 최소인 위치를 찾는다. 즉 “고정 1/3”이 아니라 **±10px 안에서 최적 정렬**을 쓰고 있다.
2. **MSE 기준 (`MSE_THRESHOLD = 500`)**
  그 최적 정렬에서 좌·우 MSE가 500 이하면 삼분할 O로 판정한다.

**저해상도(예: 640폭)에서의 논리**

- **물리적 밀림 ↔ 픽셀**  
FHD(1920)에서 7px 밀림(640 → 647)은 **전체 폭의 약 0.36%**에 해당한다.  
같은 비율을 640 폭에 적용하면 640×0.0036 ≈ **2.3px** 정도다.  
저해상도에서는 **같은 “약간 밀림”이 더 적은 픽셀 수**로 나타나고, 우리는 여전히 **±10px** 구간을 전부 탐색한다.  
따라서 640×360에서도 base=213, 탐색 구간 [203..223]이면, FHD에서의 “7px 밀림”에 해당하는 **약 2.3px**는 이 구간 안에 들어가고, **같은 정도의 밀림까지 그대로 감안한 탐색**이 가능하다.  
오히려 10px가 차지하는 **상대 폭 비율**은 640에서가 더 크므로(10/640 ≈ 1.56% vs 10/1920 ≈ 0.52%), “약간 밀린” 경우를 **상대적으로 더 넉넉하게** 커버한다고 보는 것이 맞다.
- **MSE 판정**  
저해상도 프레임은 원본의 **공간 평균(다운스케일)** 이다.  
좌·우가 원본에서 비슷하면(삼분할, 밀림 포함), 평균한 뒤에도 비슷하다.  
그래서 “좌=우인가?”라는 **질문 자체는 해상도가 낮아져도 유지**되고, MSE ≤ 500 기준으로 삼분할 여부를 보는 **논리**도 그대로 적용된다.  
다만 픽셀 수가 줄어들어 MSE 값 자체는 달라질 수 있고, **경계선상의 아주 미세한 차이**는 스케일 시 희석될 수 있다.  
즉, “선이 647/1273으로 약간 밀려 있다”는 **정렬 문제**는 위에서처럼 ±10px 탐색으로 여전히 처리 가능하고, “좌·우가 전반적으로 같은가?”라는 **유사도**는 저해상도에서도 유효하다.

**정리**

- **“약간 밀리는 경우”까지 체크 가능한가?**  
**가능하다.**  
  - 정렬: 같은 물리적 밀림이 저해상도에서는 더 적은 픽셀 수로 나타나며, ±10px 탐색이 그 범위를 포함하고, 상대적으로는 더 넓은 비율을 커버한다.  
  - 판정: 좌·우 유사도는 다운스케일 후에도 같은 논리로 적용되므로, “삼분할 + 약간 밀림”은 640×360(또는 960×540, 1280×720)에서도 논리적으로 검출 대상이다.
- **주의할 점**: 해상도를 **너무** 낮추면(예: 480 이하) 10px가 폭 대비 비율로 너무 커져, 정렬이 지나치게 성기게 될 수 있어, 실험에서는 640을 기본으로 두고 품질을 보는 것이 안전하다.

### 4.2 경계 정밀화

경계 정밀화는 **후보 구간마다** “삼분할이 시작/끝나는 시각”을 **이진 탐색**으로 찾는다. 한 번의 중간 시각 확인마다 `extract_frame`(또는 캐시 로드) 1회가 들기 때문에, **정밀도가 높을수록·구간이 길수록** 호출 수가 늘어나 시간이 오래 걸린다. 아래는 이 구간을 줄이기 위한 **접근법을 카테고리별**로 정리한 것이다.

#### BOUNDARY_TOLERANCE 동작 원리 (한 번의 탐색 시나리오)

**목표**
코스 스캔에서 "30초에는 삼분할 O, 그 이전 0초에는 X"라고 나왔다고 하자. 그러면 **"삼분할이 처음으로 나오는 시각"**은 0초와 30초 **사이** 어딘가에 있다. 경계 정밀화는 그 **정확한 시각**을 이진 탐색으로 찾는다.

**규칙 하나**

- "정확하다"의 기준이 **BOUNDARY_TOLERANCE**(**기본 1.875초**)다. 이 값은 CLI **`--boundary-tolerance SEC`** 로 실행 시 지정할 수 있다.
- **구간 길이**가 이 값보다 **클 때만** 반복하고, 구간이 **1.875초 이하로 좁혀지면** 탐색을 멈춘다.
- 즉, "시작 시각을 **1.875초 단위 정밀도**로만 알면 된다"고 보는 것이다.

> **메모**: 기본값을 0.5초에서 **1.875초**로 변경함. 경계 정밀화 반복 수 감소로 속도 이득이 크고, 실사용상 정확도 차이가 크지 않아 기본을 1.875로 통일.

**한 번의 "시작 시각" 탐색 시나리오 (숫자 예시)**  

- 탐색 구간: **t_low = 0초**, **t_high = 30초** (30초에서는 삼분할 O인 걸 알고 있음).  
- 0초 한 번 확인 → 삼분할 X라고 가정.  
- **반복**: "구간 길이 > 0.5초"인 동안만 다음을 한다.  
  - 중간 시각 **mid = (t_low + t_high) / 2** 를 정한다.  
  - **mid** 초에서 1프레임 뽑아서 삼분할인지 확인한다 (= `extract_frame` 1회 또는 캐시 1회).  
  - 결과에 따라 구간을 반으로 줄인다:  
    - mid가 삼분할 **O** → "시작 시각은 mid보다 왼쪽에 있다" → **t_high = mid**.  
    - mid가 삼분할 **X** → "시작 시각은 mid보다 오른쪽에 있다" → **t_low = mid**.
  - 이렇게 하면 매 반복마다 **구간 길이가 절반**이 된다.

**BOUNDARY_TOLERANCE = 0.5초일 때, 대략 몇 번 확인하는가**


| 단계  | t_low   | t_high   | 구간 길이          | mid         | mid 확인 결과(예시) | 다음 구간               |
| --- | ------- | -------- | -------------- | ----------- | ------------- | ------------------- |
| 0   | 0       | 30       | 30초            | (초기 0초만 확인) | X             | [0, 30] 유지          |
| 1   | 0       | 30       | 30 > 0.5       | 15          | X             | [15, 30]            |
| 2   | 15      | 30       | 15 > 0.5       | 22.5        | O             | [15, 22.5]          |
| 3   | 15      | 22.5     | 7.5 > 0.5      | 18.75       | X             | [18.75, 22.5]       |
| 4   | 18.75   | 22.5     | 3.75 > 0.5     | 20.625      | O             | [18.75, 20.625]     |
| 5   | 18.75   | 20.625   | 1.875 > 0.5    | 19.6875     | X             | [19.6875, 20.625]   |
| 6   | 19.6875 | 20.625   | 0.9375 > 0.5   | 20.15625    | O             | [19.6875, 20.15625] |
| 7   | 19.6875 | 20.15625 | **0.47 ≤ 0.5** | —           | **멈춤**        | 시작 시각 ≈ 20.16초로 확정  |


- "mid 확인"은 **1번에 extract_frame 1회**이다.  
- 위 예시에서는 **6번** 반복(매번 mid 1회 확인) + 처음 t_low 1회 = **총 7회** 프레임 추출로, "시작 시각"을 **약 20.16초**로 정했다.

**BOUNDARY_TOLERANCE를 바꾸면 어떻게 되나**  

- **0.5초** → "구간이 0.5초 이하가 될 때까지" 반복하므로, 위처럼 반복이 많이 일어난다.  
- **1.0초**로 키우면 → "구간이 1.0초 이하가 되면" 멈춘다. 그러면 **더 일찍** 멈추므로 **반복 횟수(확인 횟수)**가 줄어든다. 대신 "시작 시각"이 최대 약 ±0.5초 정도 어긋날 수 있다.  
- **정리**: BOUNDARY_TOLERANCE를 **크게** 하면 → "정밀도"를 포기하는 대신 **반복 수가 줄어들어** 경계 정밀화 시간이 줄어든다.

**구간이 매번 절반이 되는지 / 30초일 때 몇 번이면 끝나는지**  

- 네, **매 반복마다 구간 길이가 정확히 절반**이 됩니다.  
예: 30 → 15 → 7.5 → 3.75 → 1.875 → 0.9375 → **0.46875** (여기서 0.5 이하라 멈춤).  
(말씀하신 30→15→7→3.5→1.75→0.875도 같은 흐름이고, 정확히는 7.5, 3.75, 0.9375, 0.46875처럼 절반씩 줄어듭니다.)  
- **30초 구간**이고 BOUNDARY_TOLERANCE = 0.5초일 때:  
  - 반복은 **구간 길이가 0.5초 이하가 될 때까지** 하므로,  
  `30 / 2^k ≤ 0.5` 가 되는 최소 k가 반복 횟수입니다.  
  → `2^k ≥ 60` → **k = 6** (30/2^5=0.9375 > 0.5, 30/2^6 ≈ 0.47 ≤ 0.5).  
  - 따라서 **루프 안에서는 정확히 6번** mid 확인을 하고, 그 전에 t_low 한 번 확인하면 **총 7회** 프레임 추출로 한쪽 경계(시작 또는 끝)를 찾습니다.
- 한 후보 구간마다 "시작 시각" 1번 + "끝 시각" 1번을 찾으므로, **후보 하나당 최대 7×2 = 14회** 추출이 들 수 있습니다 (양쪽 모두 30초 구간인 경우).

**최대 반복 횟수를 두는 이유**  

- 구간이 매우 길면(예: 10분) "구간 길이 > 0.5"가 참인 횟수가 많아져서, 이론상 **매우 많은** mid 확인이 일어날 수 있다.  
- "최대 N회만 반복"으로 **상한**을 두면, 아무리 구간이 길어도 확인 횟수는 N 이하로 막을 수 있다.  
- 즉, **정밀도는 그대로 두고**, "극단적으로 긴 구간"에서만 반복이 무한히 늘어나는 것을 막는 것이다.

**예: 1.875초 오차까지 허용할 때 (3시간 영상 기준 이득)**  

- 지금은 **0.46875초**(≈0.5초)까지 좁혀서 한쪽 경계당 **7회** 추출(1 + 6반복).  
- **1.875초에서 멈추도록** 하려면 `BOUNDARY_TOLERANCE = 1.875`로 두면 된다. 그러면 `30/2^k ≤ 1.875` → `2^k ≥ 16` → **k = 4**만 반복하면 되므로, 한쪽 경계당 **5회** 추출(1 + 4반복).  
- **한쪽당 7회 → 5회**이므로, 경계 정밀화에서 쓰는 **추출 횟수가 약 29%(2/7) 줄어든다.**  
- 3시간 영상이면 후보 구간이 대략 **8~12개** 정도라고 가정하면, 후보당 양쪽 경계라서 **후보당 4회 × 후보 수**만큼 추출이 줄어든다. (예: 10개 후보면 10×4 = **40회** 추출 감소.)  
- 추출 1회를 네트워크 기준 대략 0.1~~0.2초로 잡으면, **40회 × 0.15초 ≈ 6초** 정도 경계 정밀화 구간에서 단축될 수 있다. (실제는 환경마다 다르지만, “경계 단계가 10초대면 3~~4초 줄어든다” 수준으로 보면 된다.)

#### 1) 정밀도·반복 제어 (정확도 ↔ 속도 트레이드오프)

**현재 동작**  

- `BOUNDARY_TOLERANCE = 0.5`(초). 이진 탐색은 `t_high - t_low > 0.5`인 동안만 반복하고, 한 번 반복할 때마다 **중간 시각 1회**에 대해 프레임 추출(또는 캐시 조회)을 한다.  
- 구간 길이 30초라면 시작/끝 각각 이진 탐색이므로, 대략 `2 × ceil(log2(30/0.5))` ≈ 12회 수준의 확인이 한 후보당 필요하고, 후보가 9개면 **약 100회 이상**의 추출이 경계 단계에서 일어날 수 있다.

**접근 1: 정밀도 완화**  

- `BOUNDARY_TOLERANCE`를 **1.0초** 등으로 키우면, “0.5초 단위”보다 일찍 탐색을 멈추므로 **반복 횟수가 줄어들고** 그만큼 추출 횟수도 감소한다.  
- 대신 **경계 위치 정확도**는 떨어진다(시작/끝이 최대 약 ±0.5초 정도 어긋날 수 있음).  
- **적용 방법**: CLI `--boundary-tolerance SEC`로 지정 가능 (미지정 시 코드 기본값 **1.875초**). 더 정밀하게 하려면 `--boundary-tolerance 0.5` 등으로 줄이면 된다.

**접근 2: 이진 탐색 반복 상한**  

- 정밀도(0.5초)에 도달하기 전에 **최대 반복 횟수**(예: 10회)를 두고, 그만큼만 확인한 뒤 종료한다.  
- 매우 긴 후보 구간에서만 반복이 많이 늘어나는 것을 막아, **최악의 경우** 호출 수를 제한한다.  
- 정밀도는 그대로 두고, “극단적으로 긴 구간”에서만 정밀도를 포기하는 효과.

**요약**: 경계 정밀화 시간을 줄이려면 **정밀도 완화** 또는 **반복 상한**으로 “확인 횟수”를 줄이는 것이 첫 번째 손댈 지점이다.

#### 2) 캐시 활용 (경계 단계에서 ffmpeg 호출 일부 생략)

**현재 동작**  

- 경계 이진 탐색에서 필요할 때마다 `extract_frame(path, t, ...)`를 호출한다.  
- `--coarse-cache`를 쓰고 **캐시가 원본 해상도**일 때만, 요청 시각 `t`가 **캐시에 있는 시각과 0.5초 이내**이면 디스크에서 캐시를 읽어 쓰고 ffmpeg 추출은 하지 않는다.

**언제 이점이 있는가**  

- **원본 해상도**로 코스 스캔을 했을 때만 경계에서 캐시를 쓴다. (`--coarse-scale` 없이 `--coarse-cache`만 사용.)  
- 캐시 폴더는 **로컬 디스크**에 두는 것이 좋다. 네트워크 경로에 캐시를 두면 코스 단계 쓰기 비용이 커져 전체가 느려질 수 있다.  
- 이진 탐색이 **우연히 코스 시점(0, 30, 60, …) 근처**를 요청할 때만 캐시 적중이 나오므로, 적중 비율에 비례해서 경계 단계 시간이 줄어든다.

**저해상도와 함께 쓸 때**  

- `--coarse-scale 640 --coarse-cache`처럼 **저해상도로만** 캐시하면, 경계에서는 **캐시를 사용하지 않고** 항상 원본 해상도로 추출한다. 따라서 이 조합에서는 경계 단계 최적화 이점이 없다.

**요약**: 경계 정밀화를 가볍게 하려면 **원본 해상도 코스 + 로컬 캐시**로 “경계에서의 재사용”만 노리는 방식이 해당한다.

#### 3) 경계 단계 저해상도(실험·선택)

**아이디어**  

- 경계 정밀화에서도 **640×360** 등 저해상도로 `extract_frame`을 호출하고, 그 프레임으로 삼분할 여부만 판정한다.  
- 호출 1회당 전송·연산량이 줄어들어 **경계 단계 전체 시간**은 줄어들 수 있다.

**트레이드오프**  

- 경계 **시각 정확도**가 떨어질 수 있다. “시작/끝”을 찾는 기준이 저해상도 프레임이 되므로, 원본 해상도로 찾았을 때와 최대 약 0.5초 단위 차이가 날 수 있다.  
- 현재 구현은 **경계는 항상 원본 해상도**를 쓰도록 되어 있어, 이 방식을 쓰려면 옵션(예: `--boundary-scale 640`)을 새로 두고, 경계용 `extract_frame`에만 scale을 넘기는 식으로 구현해야 한다.

**요약**: “경계도 저해상도 허용”은 **실험용**으로만 고려할 만하고, 정확도가 중요하면 1)·2)를 먼저 적용하는 편이 안전하다.

### 4.3 병합(merge_segments) 최적화

**전제 조건**: 입력 영상은 **반드시 네트워크 공유 경로**에서만 접근·편집하는 상황이다. 원본을 로컬로 복사해 쓰는 것은 불가하다.

---

#### 병합 단계가 오래 걸리는 이유

| 원인 | 설명 |
|------|------|
| **입력이 네트워크** | 모든 구간 추출 시 ffmpeg가 **네트워크 경로의 원본**을 읽는다. 구간 수·길이만큼 순차적으로 읽기 때문에 대역폭·지연이 그대로 누적된다. |
| **구간 순차 처리** | 구간 1 → 구간 2 → … 순서로만 처리한다. N개 구간이면 N번의 “네트워크 읽기 + 쓰기”가 **겹치지 않고** 이어져 총 시간이 선형으로 늘어난다. |
| **임시·출력도 네트워크일 수 있음** | 현재 `tmpdir`은 **출력 파일의 부모 폴더** 아래에 만든다. `--merge`로 네트워크 경로를 주면 임시 TS 파일들도 네트워크에 쓰고, concat 시 네트워크에서 읽어서 다시 네트워크에 쓴다. **쓰기 2회(구간별 + concat)** 모두 네트워크 I/O가 된다. |
| **concat 단일 대용량 쓰기** | 마지막에 segment_0001.ts … 을 한 MP4로 합친다. 이때 **한 파일 전체**를 쓰므로, 출력이 네트워크면 concat 단계만으로 수십 초가 걸릴 수 있다(예: 51초). |
| **구간당 2단계 ffmpeg** | 각 구간마다 ① `-ss -t -c copy`로 잘라 temp → ② ffprobe 키프레임 → ③ 첫 키프레임부터 재절단해 segment.ts. 디코딩은 없어도 **읽기/쓰기·프로세스 기동**이 구간당 2회씩 발생한다. |

정리하면, **읽기(원본)·쓰기(임시·최종)** 가 모두 네트워크에 걸쳐 있고, **순차 처리**라서 병목이 겹쳐지지 않는다.

---

#### 최적화 방안 리스트 (입력은 네트워크 고정인 경우)

| # | 방안 | 논리 / 어떻게 시간이 줄어드는지 | 비고 |
|---|------|--------------------------------|------|
| **1** | **출력·임시를 로컬로** | `--merge`로 **로컬 경로**(예: `C:\temp\merged.mp4`)를 지정하고, 코드에서 `tmpdir`도 **로컬**(예: `output_path.parent` 또는 시스템 임시 디렉터리)에 두면, **쓰기**는 전부 로컬에서만 일어난다. 원본 읽기는 여전히 네트워크이지만, concat 시 “네트워크에서 여러 TS 읽기 + 네트워크에 MP4 쓰기”가 **로컬에서 TS 읽기 + 로컬에 MP4 쓰기**로 바뀌어 concat 시간이 크게 줄고, 구간별 임시 쓰기 지연도 사라진다. | 입력은 네트워크 유지. 결과만 필요 시 수동으로 네트워크로 복사. |
| **2** | **구간별 병렬 추출** | 구간 1, 2, … N을 **동시에** (ProcessPoolExecutor 또는 스레드로) ffmpeg cut 실행. 원본이 네트워크이므로 동시에 여러 스트림을 읽게 되어 **대역폭이 허용하는 범위에서** 총 경과 시간이 “가장 긴 구간 1개 + α”에 가까워진다. 순차면 47초+43초=90초 → 병렬이면 max(47,43)+오버헤드 수준으로 줄 수 있다. | 동시 접속 수가 너무 크면 NAS/공유기 부하로 오히려 느려질 수 있음. 워커 수 2~4 등으로 제한해 실험. |
| **3** | **임시 디렉터리를 로컬로 고정** | `tmpdir`을 **항상** 로컬(예: `tempfile.gettempdir()` 또는 `--merge`로 준 출력이 로컬일 때만 `output_path.parent`)으로 두고, concat 결과만 `output_path`에 쓰기. 출력을 네트워크로 줄지라도 **중간 TS 파일**은 로컬에 두면, concat 단계는 “로컬에서 TS 읽기 + output_path에 1회 쓰기”가 되어 concat 읽기 쪽 지연이 사라진다. | 구현: `tmpdir` 인자/옵션 또는 환경 변수로 로컬 경로 지정. |
| **4** | **키프레임 트림 1회화 유지** | 구간당 ffprobe 1회 + 트림 ffmpeg 1회 구조는 이미 1회씩만 사용 중이다. 같은 조각에 ffprobe를 두 번 부르는 등 **중복 제거**만 점검하면 된다. | 현재 코드에서 중복 없으면 추가 절감은 적음. |
| **5** | **concat 출력만 로컬, 이후 복사** | 최종 결과를 로컬에 만든 뒤, 별도 단계(robocopy, xcopy 등)로 네트워크 공유로 복사한다. “네트워크에 큰 파일 1개 쓰기”를 **한 번**으로 묶고, 병합 스크립트 자체는 로컬 쓰기만 하여 concat 시간을 짧게 유지한다. | 사용자 워크플로: `--merge D:\out\merged.mp4` 실행 후 수동으로 공유 폴더로 복사. |
| **6** | **긴 구간 우선 또는 순서 재배치** | 이론상 “긴 구간을 먼저 처리하면 나머지가 상대적으로 빨리 끝나서 체감이 나을 수 있음”이지만, 순차 처리 자체가 바뀌지 않으면 총 시간 합은 동일하다. **병렬 추출**과 결합할 때 “긴 구간이 있는 쪽을 먼저 스케줄”하는 정도의 효과만 기대. | 단독으로는 큰 이득 없음. |

**요약 (입력 = 네트워크 고정)**

- **반드시 추천**: **출력·임시를 로컬로** (1, 3, 5). 원본은 네트워크여도 **쓰기**를 로컬로 몰면 concat·구간 쓰기 지연이 크게 줄어든다.
- **구현 이득 큼**: **구간별 병렬 추출** (2). 워커 수를 2~4 등으로 제한해 네트워크 부하를 보면서 적용.
- **유지·점검**: 키프레임 ffprobe/트림은 구간당 1회만 쓰도록 유지 (4).

#### 시나리오: 1Gbit LAN + NAS(7200 RPM HDD), 로컬 NVMe 4.0 SSD

**가정**

- **네트워크**: 같은 공유기 내 1Gbit 랜, 실측 약 **100 MB/s**.
- **제어 대상(원본 위치)**: 네트워크 공유 스토리지 = **7200 RPM HDD**. 순차 읽기 ~100–150 MB/s, 동시 다중 스트림 시 seek 증가로 실효 속도 하락.
- **입력**: 반드시 네트워크 경로 고정(원본 복사 불가).
- **로컬**: **NVMe 4.0 SSD** 사용 중(순차 읽/쓰 수 GB/s급).

**병목 관점**

- **읽기**: 원본은 네트워크 HDD → 링크·HDD 중 느린 쪽이 병목. 1Gbit로 100 MB/s 나오면 링크는 꽉 찬 상태. HDD가 순차로만 읽을 때는 비슷한 수준이므로, **동시에 여러 구간을 읽으면** HDD가 seek로 효율이 떨어져 전체가 느려질 수 있음.
- **쓰기**: 로컬이 NVMe면 구간 TS·concat 결과를 **로컬 SSD에만** 쓰면 쓰기 지연은 거의 무시 가능. 반대로 임시·출력을 네트워크(HDD)에 쓰면 100 MB/s 쓰기 + seek로 concat 단계가 수십 초 걸림.

**로컬(NVMe) 자원을 써서 속도를 올리는 방법 — 효과 큰 순서**

| 순서 | 조치 | 효과·논리 | 비고 |
|------|------|-----------|------|
| **1** | **임시·출력을 전부 로컬 NVMe로** | 구간 cut 시 “네트워크에서 읽기 → **로컬 NVMe에 쓰기**”, concat 시 “**로컬 NVMe에서 TS 읽기** → **로컬 NVMe에 MP4 쓰기**”. 쓰기·concat 읽기가 NVMe 속도로 처리되어, **concat 구간이 51초 → 수 초 이하**로 줄고, 구간별 임시 쓰기 지연도 사라짐. 읽기(원본)는 여전히 네트워크이지만, 병목이 “읽기 1종”으로만 남음. | `--merge`를 로컬 경로(예: `D:\merge_out\merged.mp4`, D: = NVMe)로 지정하고, `tmpdir`도 같은 드라이브 또는 `%TEMP%`(NVMe)에 두도록 구현. |
| **2** | **구간별 병렬 추출(워커 2명)** | 원본이 7200 RPM HDD이므로 **동시 스트림을 2개**만 둠. 순차면 구간1 읽기 → 구간2 읽기 순서로만 진행되나, 2-way 병렬이면 구간1·구간2 읽기가 겹쳐서 **총 경과 시간이 max(구간1, 구간2) + α**에 가까워짐. 1Gbit/100 MB/s를 두 스트림이 나눠 써도 HDD가 순차에 가깝게 두 스트림을 처리할 수 있는 범위. | 워커 3~4는 HDD seek 증가로 오히려 느려질 수 있음. **2**로 고정 후 측정, 필요 시 3까지만 시험. |
| **3** | **concat 후 결과만 네트워크로 복사** | 병합 결과는 로컬 NVMe에만 두고, 필요 시 **robocopy / xcopy** 등으로 네트워크 공유로 **한 번만** 복사. 병합 스크립트는 “네트워크에 큰 파일 쓰기”를 하지 않으므로 concat 시간이 짧게 유지되고, 복사는 백그라운드/수동으로 처리 가능. | 워크플로: `--merge D:\out\merged.mp4` 실행 → 완료 후 공유 폴더로 복사. |
| **4** | **임시 디렉터리를 NVMe로 고정** | `tmpdir`을 환경 변수나 옵션으로 **NVMe 드라이브**(예: `D:\tripartite_tmp`)에 고정. 출력 경로를 네트워크로 줘도 **중간 TS는 NVMe**에만 두면 concat 읽기가 NVMe에서만 일어나 concat 단계가 빨라짐. | 구현: `TRIPARTITE_TMPDIR=D:\tripartite_tmp` 또는 `--merge-tmpdir` 같은 옵션. |
| **5** | **병렬 워커 3 이상은 비추** | 7200 RPM HDD는 동시 3~4개 스트림에서 seek가 늘어나 **전체 처리량이 100 MB/s 미만**으로 떨어질 수 있음. 로컬 NVMe는 여유 있지만, **원본 읽기**가 HDD에 묶여 있으므로 워커를 2(또는 최대 3)로 두는 것이 안전. | 4 이상은 제외. |

**이 환경에서의 요약**

- **가장 효과 큼**: 임시·출력을 **전부 로컬 NVMe**로 (1, 4). concat과 구간 쓰기 지연이 거의 사라짐.
- **그다음**: 구간 추출 **2-way 병렬** (2). HDD 부담을 크게 늘리지 않으면서 경과 시간 단축.
- **마지막**: 결과는 NVMe에 두고, 필요할 때만 네트워크로 한 번 복사 (3).

#### 병합 로그로 보이는 병목 (예: 17구간, 36분 분량, 총 112초)

| 구간 | 예시 소요 | 병목 의미 |
|------|-----------|-----------|
| **최종 파일 복사** | **73.7초 (66%)** | 로컬 → 네트워크 1회 쓰기. 병합 결과 파일 크기가 크면 이 시간이 지배적. |
| **구간 추출(병렬 2)** | **~36초 (32%)** | 네트워크에서 읽기 + 로컬에 쓰기. cut(네트워크) vs trim(로컬) 비중은 구간별 로그 `cut X.X초 trim X.X초`로 확인 가능. |
| **concat 합치기(로컬)** | **2.7초 (2%)** | 로컬 NVMe 읽기/쓰기라 이미 최소. |

**정리**: 지금 로그만으로도 **1) 최종 복사**, **2) 구간 추출** 순서로 병목이 보인다. 구간별로 `cut` / `trim` 시간을 나눠 로깅하면, 구간 추출 안에서 “네트워크 읽기(cut)” vs “로컬 트림(trim)” 중 어디가 더 오래 걸리는지 구간·길이별로 확인할 수 있다.

#### 다음에 트라이해볼 것 (우선순위)

| 순서 | 항목 | 내용 | 비고 |
|------|------|------|------|
| 1 | **출력을 로컬만 쓰기** | `--merge`를 로컬 경로로 주고, 네트워크로는 사용자가 나중에 복사. 복사 73초를 병합 실행에서 제외. | 체감 최대. |
| 2 | **구간별 cut/trim 로깅** | 각 구간에 `cut X.X초 trim X.X초` 출력. cut이 대부분이면 병목은 네트워크 읽기; trim이 크면 로컬 I/O나 ffmpeg. | 구현됨. |
| 3 | **병렬 워커 3 실험** | 7200 RPM HDD 환경에서 워커 3으로 한 번 돌려 보고, 총 시간이 줄면 유지·옵션화. 늘면 2로 유지. | CLI 옵션화 시 `--merge-workers 2` 기본. |

| 5 | **concat 포맷** | MP4 단일로 concat 시 재인코딩 없이 유지. 이미 `-c copy`이면 추가 시도 여지 거의 없음. | 현재 유지. |

### 4.4 전체 워크플로

- **--segments-out 후 편집 → --segments-in --merge** 흐름은 이미 최적: 한 번만 검출하고, 잘라낼 구간만 골라서 병합하므로 재검출 비용이 없음.
- **검출 시 --merge까지 한 번에** 하지 않고, 목록만 저장해 두고 사용자가 편집한 뒤 병합하는 현재 방식 유지 권장.

### 4.5 기타

- **CuPy**:  
`--cuda` 시 CuPy가 있으면 판정 연산을 GPU로 할 수 있음. 이미 사용 중이면 유지; 코스 스캔 병목은 주로 **디코딩·I/O**이므로, GPU 디코딩(ffmpeg CUDA)이 더 중요.
- **네트워크 I/O**:  
원본이 네트워크일 때, 로컬 복사본으로 검출·병합을 돌리면 가장 빠르다. **단, 반드시 네트워크 공유 경로에서만 편집·실행해야 하는 상황**이라면 원본 복사는 불가하므로, §4.3처럼 **출력·임시만 로컬**로 두고 병렬 추출을 적용하는 식으로 최적화한다.
