# CUDA 12.8 vs 13.0 조사: tri-split-finder 관점

현재 **CUDA 12.8** 사용 중인 환경에서 **CUDA 13.0** 업그레이드 필요 여부와, 13.0에서 기대할 수 있는 성능 최적화 가능성을 정리했습니다.

---

## 1. 프로젝트에서 CUDA를 쓰는 곳

| 구간 | 용도 | 사용 방식 |
|------|------|-----------|
| **ffmpeg** | GPU 디코딩 (NVDEC) | `-hwaccel cuda` → 드라이버/런타임 수준 |
| **CuPy** | 프레임 crop / 3등분 slice / MSE 계산 | `cupy-cuda12x`, `_is_tripartite_frame_gpu()` |

- **판정 연산**: `cp.asarray()` → 슬라이스 → `astype(cp.float64)` → `(left - right)**2` → `cp.mean(...).item()`  
- **사용하는 연산**: 메모리 복사·슬라이스, 원소별 뺄셈·제곱, **reduction(mean)**  
- **사용하지 않는 것**: cuBLAS(GEMM), cuFFT, cuSPARSE, 커스텀 CUDA 커널

---

## 2. CUDA 12.8 vs 13.0 요약

### 2.1 CUDA 12.8에서 이미 있는 것

- Blackwell 아키텍처 지원
- CUDA Graphs 개선(조건부 노드 등)
- cuBLAS/cuSPARSE/cuSOLVER/nvJPEG 등 Blackwell 대응

### 2.2 CUDA 13.0에서 추가·변경된 것 (관련 있는 부분만)

| 항목 | 내용 | 이 프로젝트와의 관계 |
|------|------|----------------------|
| **Tile 프로그래밍 모델** | SIMT 외에 타일 단위 모델 기반 마련 | Python/NumPy 스타일 배열 연산의 미래 확장. **현재 CuPy API로는 직접 활용 불가**. |
| **벡터 타입 32바이트 정렬** | `double4` 등 → `double4_32a`로 Blackwell에서 256비트 로드/스토어 활용 | **커스텀 커널을 쓰지 않으면** 우리 코드에서 직접 쓸 일 없음. CuPy 내부가 활용하면 이론상 reduction 등에서 소폭 이득 가능. |
| **cuBLAS** | `CUBLAS_GEMM_AUTOTUNE`, L3 커널(SYRK, HERK 등) FP32/CF32 개선 | **GEMM/선형대수 미사용** → 판정 파이프라인에 직접 해당 없음. |
| **cuFFT** | 단정밀 C2C 다차원 FFT 등 성능 개선 | **FFT 미사용** → 해당 없음. |
| **fatbin 압축 (ZStd)** | 바이너리 크기 감소(기본 balance 모드) | 빌드/배포 크기 이득. **실행 시간과는 무관**. |
| **CCCL 3.0** | Thrust/CUB/libcudacxx 통합, C++17, 헤더 경로 변경 | CuPy 등이 13.0으로 빌드할 때 내부적으로만 영향. 우리는 CuPy wheel만 사용. |
| **플랫폼 통합** | Arm(예: DGX Spark, Jetson Thor) 단일 툴체인 | x86 Windows 환경에서는 해당 없음. |
| **Feature-complete** | sm 7.5 미만(튜링 이전) 오프라인 컴파일 제거 | 최신 GPU만 쓴다면 문제 없음. |

---

## 3. 결론: 13.0이 “꼭” 필요한가?

**아니요. 현재 기능만 놓고 보면 CUDA 12.8로 충분합니다.**

- **ffmpeg**: NVDEC는 드라이버가 지원하는 CUDA 버전만 맞으면 되며, 12.8 드라이버로 12.x 런타임 사용에 문제 없음.
- **CuPy**: `cupy-cuda12x`가 12.8과 호환되며, 13.0 전용 wheel(예: `cupy-cuda13x`)은 현재 일반적으로 제공되지 않음.  
  따라서 **지금 단계에서는 12.8 + cupy-cuda12x 유지가 현실적인 선택**입니다.

---

## 4. 13.0으로 올렸을 때 “우리 코드”에서 기대할 수 있는 것

우리 워크로드는 **이미지 영역 crop → 좌/우(또는 좌/중/우) MSE**가 전부라, 13.0의 주력 개선 대상과 직접 겹치지 않습니다.

| 13.0 기능 | 우리 코드에서의 활용 가능성 |
|-----------|----------------------------|
| **Tile 모델** | 장기적으로 Python/배열 API로 제공될 수 있으나, **현재 CuPy API로는 사용 불가**. 미래에 CuPy가 tile 기반 백엔드를 쓰면, 그때 재평가 가능. |
| **32바이트 정렬 벡터 타입** | 커스텀 커널을 작성하지 않으므로 **직접 사용 불가**. CuPy가 reduction 등을 13.0용으로 최적화하면 **간접적으로 소폭 이득** 가능. |
| **cuBLAS/cuFFT** | MSE는 원소별 연산 + sum/mean 뿐이라 **해당 없음**. |
| **ZStd 압축** | 실행 성능과 무관. |

정리하면, **13.0으로 올려도 “이 프로젝트 코드를 13.0 전용으로 최적화할 만한 구체적인 API”는 거의 없고**,  
가능한 이득은 **CuPy가 13.0 툴킷으로 빌드된 wheel을 제공한 뒤, 그걸 썼을 때의 간접적인 최적화(라이브러리 내부)** 정도로 보는 것이 맞습니다.

## 5. 현재 구조에서 “성능”을 더 내고 싶다면 (12.8이든 13.0이든 공통)

`PIPELINE_AND_OPTIMIZATION.md`와 동일한 방향이지만, GPU 관점만 짧게 정리하면:

1. **병목이 디코딩·I/O인 경우**  
   - `--cuda`로 ffmpeg GPU 디코딩은 이미 사용 중.  
   - 추가 이득은 **저해상도 코스 스캔**(`--coarse-scale 640` 등), **경계 정밀화 완화**(`--boundary-tolerance`), **캐시 전략**(원본 해상도 + 로컬 캐시) 등 **알고리즘/옵션 조정**에서 나옵니다.  
   - 여기는 CUDA 12.8 vs 13.0과 무관합니다.

2. **GPU 판정(CuPy) 구간을 줄이고 싶은 경우**  
   - 프레임 단위로 `cp.asarray` → `mean` → `.item()` 호출이 반복되므로, **프레임 수 자체를 줄이는 것**(코스 간격, 저해상도, 경계 탐색 횟수)이 가장 효과적입니다.  
   - 12.8이든 13.0이든, “MSE 한 번 계산” 자체를 13.0만의 기능으로 크게 가속할 여지는 제한적입니다.

3. **나중에 CuPy가 13.0 wheel을 제공할 때**  
   - `cupy-cuda13x` 등이 나오면, **동일 입력·동일 옵션으로 12.x vs 13.x 벤치마크**만 해보면 됩니다.  
   - 기대치는 “수 % 이내의 차이” 정도로 두는 것이 안전합니다.

## 6. 요약 표

| 질문 | 답 |
|------|----|
| **지금 13.0이 꼭 필요한가?** | **아니요.** 12.8 + cupy-cuda12x로 현재 기능 모두 사용 가능. |
| **13.0으로 올리면 우리 코드가 빨라지나?** | **직접적으로는 거의 없음.** cuBLAS/cuFFT/Tile/벡터 타입을 우리가 직접 쓰지 않음. |
| **13.0에서 우리가 “최적화”할 수 있는 부분이 있나?** | **코드 레벨에서는 거의 없음.** CuPy가 13.0용으로 내부 최적화한 wheel을 쓰는 것이 유일한 실질적 경로. |
| **지금 할 만한 일** | 12.8 유지 + `--coarse-scale`, `--boundary-tolerance`, 병합 시 로컬 출력·병렬 등 **기존 문서의 최적화 권장** 적용. |

이 문서는 2025년 3월 기준 공개된 CUDA 13.0 블로그·릴리스 노트와, `tripartite_section_check.py` 및 Docs 내용을 바탕으로 작성했습니다.
