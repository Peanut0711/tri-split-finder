# 코스 스캔(Coarse Scan) 개요

삼분할 구간 검출기(`tripartite_section_check.py`)에서 **코스 스캔**이 무엇인지, 기존 방식의 한계, 이번 수정 내용, 기대 효과와 사이드 이펙트를 정리한 문서입니다.

---

## 1. 코스 스캔 원리

### 1.1 목적

장편 영상(3~6시간) 전체를 **프레임 단위로** 삼분할 여부를 검사하면 시간이 너무 오래 걸립니다.  
그래서 먼저 **일정 간격(기본 30초)의 시점만 샘플링**해 삼분할 여부를 판정하고, 그 결과를 바탕으로 **삼분할로 보이는 구간의 시작·끝만 이진 탐색으로 정밀화**하는 2단계 구조를 사용합니다.

- **1단계: 코스 스캔** — 0초, 30초, 60초, … 처럼 `coarse_interval_sec` 간격의 시점에서 1프레임씩 뽑아 삼분할 여부만 O/X로 기록.
- **2단계: 경계 정밀화** — 코스 결과에서 연속으로 O가 나온 구간을 후보로 삼아, 구간의 **시작 시점**과 **끝 시점**만 이진 탐색으로 정밀하게 찾음.

즉, 코스 스캔은 **“어디 근처에 삼분할 구간이 있는지”**를 빠르게 좁혀 주는 역할을 합니다.

### 1.2 시점 목록

- 영상 길이 `duration`, 간격 `coarse_interval_sec`(기본 30초)으로  
  `t = 0, 30, 60, …` 와 같이 시점 리스트 `ts`를 만듭니다.
- 끝부분이 빠지지 않도록 `duration - 0.1` 정도의 시점도 한 번 추가합니다.
- 예: 3시간 36분 영상 + 30초 간격 → 약 435개 시점.

### 1.3 각 시점에서 하는 일

각 시점 `t`에 대해:

1. **프레임 추출**  
   ffmpeg로 `t` 초 지점의 1프레임을 추출합니다.  
   옵션에 따라 `--coarse-scale`(640/960/1280)으로 저해상도 추출 가능.
2. **삼분할 판정**  
   추출한 프레임의 높이 중앙부(상단 팝업 제외)만 잘라, 좌/우(또는 좌/중/우) 영역의 픽셀 MSE를 계산해  
   `MSE_THRESHOLD` 이하면 삼분할 O, 아니면 X로 기록.

결과는 `(시점, 삼분할 여부)` 리스트가 되고, 이걸 이용해 “연속으로 O인 구간”을 후보로 수집한 뒤 경계 정밀화로 넘깁니다.

### 1.4 파이프라인 내 위치

```
메타(ffprobe) → 시점 목록 생성 → [코스 스캔] → 코스 정렬 → 후보 구간 수집 → 경계 정밀화 → 병합 → 최소 길이 필터
```

실제 로그에서는 **코스 스캔이 전체 소요 시간의 대부분(예: 95% 이상)**을 차지하는 경우가 많습니다.

## 2. 기존 코스 스캔 방식과 아쉬운 점

### 2.1 기존 방식

- **시점별 1:1 ffmpeg 호출**  
  샘플 시점이 435개면 **ffmpeg를 435번** 실행했습니다.
- 각 호출마다:
  - `-ss <시점> -i <파일> -vframes 1 ...` 형태로 **해당 시점으로 시크** 후 1프레임만 디코딩
  - (병렬 사용 시) `ProcessPoolExecutor`로 여러 프로세스가 동시에 **서로 다른 시점**으로 ffmpeg 실행

즉, “시점 수 = ffmpeg 실행 횟수”였습니다.

### 2.2 아쉬운 점

1. **시크 비용**  
   장편 영상에서 매번 **랜덤 시크**를 하면, ffmpeg가 키프레임을 찾고 그 지점부터 디코딩해야 해서 호출당 비용이 큽니다.  
   시점 수가 많을수록 이 비용이 그대로 곱해집니다.

2. **I/O 부담**  
   입력이 **네트워크 경로**(예: `\\192.168.8.113\...`)일 때, 수백 번의 서로 다른 시점 접근은  
   순차 읽기보다 훨씬 불리하고, 병렬로 돌리면 네트워크/디스크가 동시 접근에 시달립니다.

3. **프로세스/호출 오버헤드**  
   ffmpeg를 435번 부르면 프로세스 생성·종료와 파이프/버퍼 처리 비용이 435번 반복됩니다.

4. **실제 체감**  
   예: 3시간 36분 영상, 435 시점, workers 16, CUDA + coarse-scale 640 사용 시  
   **코스 스캔만 1분 25초** 정도 걸리고, 전체 시간의 약 95%를 차지하는 경우가 있었습니다.

## 3. 이번에 수정한 기능

### 3.1 개요

**“시점당 ffmpeg 1회”** 대신 **“담당 구간당 ffmpeg 1회”**로 바꿔,  
한 번의 ffmpeg가 **구간을 연속 재생하듯 디코딩**하면서 **일정 간격(예: 30초)의 프레임만** 받아오도록 했습니다.

- **기존**: 시점 435개 → ffmpeg **435번**
- **수정 후**(workers 16 기준): 시점을 16개 구간으로 나누고, 구간당 1번씩만 ffmpeg 실행 → ffmpeg **16번**

### 3.2 추가/변경된 요소

#### (1) `extract_frames_batch()`

- **역할**: 구간 `[start_t, end_t]`를 **한 번의 ffmpeg**로 열고, `interval_sec`(예: 30초) 간격으로 나온 프레임만 받아서 반환.
- **방식**  
  - `-ss start_t -i <파일> -t <구간 길이> -vf "fps=1/<interval_sec>"` (필요 시 `scale=...` 추가)  
  - 한 스트림에서 **연속 디코딩**하므로 시크는 구간 시작 한 번만 발생.
- **반환**: `[(t, frame), ...]` — 시점과 BGR 프레임 쌍 리스트.

이렇게 하면 “구간 전체를 재생하다가 30초마다 한 장씩만 취하는” 형태가 되어, 시크·디코딩이 한 번에 묶입니다.

#### (2) `_coarse_worker_batch()`

- **역할**: 한 워커가 담당하는 **시점 리스트(한 구간)**에 대해:
  1. `extract_frames_batch()`로 그 구간의 30초 간격 프레임을 **한 번에** 추출.
  2. 구간 내 각 시점에 대해 해당 프레임이 있으면 사용, 없으면(예: 마지막 시점이 30초 단위가 아닌 경우) 기존처럼 `extract_frame()` 1회로 보정.
  3. 각 프레임에 대해 `is_tripartite_frame()`로 삼분할 판정.
  4. `--coarse-cache` 사용 시 기존과 동일하게 캐시 저장.
- **반환**: `[(t, is_tripartite), ...]`

기존 `_coarse_worker()`는 “시점 하나 → ffmpeg 1회 → 판정”이었는데,  
배치 워커는 “시점 N개 → ffmpeg 1회(배치) → N번 판정”으로 바뀐 것입니다.

#### (3) `find_segments()` 내부 분기

- **`n_workers <= 1`**  
  기존과 동일: 시점마다 `extract_frame()` 한 번씩 호출하는 순차 처리 유지.
- **`n_workers > 1`**  
  시점 리스트 `ts`를 워커 수만큼 **구간(chunk)**으로 나누고,  
  각 구간에 대해 `_coarse_worker_batch` 한 번씩만 호출.  
  각 워커는 자신의 구간에 대해 위 (2)처럼 배치 추출 후 판정만 수행.

진행률 출력(30개마다 등)은 기존과 같은 주기로 유지됩니다.

### 3.3 사용 방법

코드/CLI 변경 없이, **`--workers`를 2 이상**으로 두면 자동으로 배치 모드가 사용됩니다.

```bash
python tripartite_section_check.py input.ts --cuda --workers 16 --coarse-scale 640 --segments-out
```

`--workers 1` 또는 `0`이면 예전처럼 시점당 1회 ffmpeg 방식이 그대로 동작합니다.

## 4. 기대 효과와 사이드 이펙트

### 4.1 기대 효과

1. **ffmpeg 호출 수 감소**  
   시점 435개, workers 16 → ffmpeg **435회 → 16회** 수준으로 줄어,  
   프로세스/파이프 오버헤드와 시크 횟수가 크게 감소합니다.

2. **시크·I/O 효율**  
   구간당 **한 번만 시크**하고 나머지는 **순차 디코딩**이므로,  
   랜덤 시크가 많은 기존 방식보다 디코더와 I/O에 유리합니다.

3. **네트워크/공유 경로**  
   같은 네트워크 경로라도, 동시에 여러 시점을 요구하는 대신  
   “구간별 순차 읽기”가 늘어나서 I/O 경합이 완화될 수 있습니다.

4. **체감 시간**  
   동일 조건(3시간 36분, 30초 간격, workers 16, CUDA, coarse-scale 640)에서  
   코스 스캔 구간이 **이전 대비 상당히 단축**될 수 있습니다(실제 수치는 환경에 따라 다름).

### 4.2 사이드 이펙트·주의점

1. **마지막 시점이 30초 단위가 아닌 경우**  
   코스 시점 리스트에 `duration - 0.1` 같은 값이 들어가면,  
   해당 구간 배치에서는 “30초 간격” 프레임만 나오므로 그 시점에 대한 프레임이 없을 수 있습니다.  
   이 경우 `_coarse_worker_batch` 내부에서 **해당 시점만 기존처럼 `extract_frame()` 1회**로 보정하므로,  
   결과의 정확도는 유지되고, 마지막 한 시점만 예전 방식으로 처리됩니다.

2. **워커당 구간 길이**  
   구간이 길수록 한 번의 ffmpeg가 디코딩하는 시간이 길어집니다.  
   타임아웃은 `extract_frames_batch` 안에서 `duration`에 비례해 설정되어 있어,  
   매우 긴 영상에서도 구간이 나뉘어 있으므로 극단적으로만 걸리면 됩니다.

3. **메모리**  
   한 워커가 “구간 내 30초 간격 프레임 전체”를 한꺼번에 메모리에 올립니다.  
   저해상도(예: 640×360)와 보통 워커 수(예: 16)에서는 부담이 크지 않지만,  
   워커 수를 매우 크게 하면 구간당 시점 수는 줄고, 반대로 워커를 너무 적게 하면 구간당 프레임 수가 늘어납니다.

4. **순차 모드(`n_workers <= 1`)는 미변경**  
   워커 1 이하일 때는 여전히 시점당 1회 ffmpeg입니다.  
   단일 프로세스에서도 “한 ffmpeg가 구간을 쭉 읽고 30초마다 프레임만 주는” 방식으로 확장하는 것은 가능하나,  
   이번 수정에서는 병렬 배치만 적용했습니다.

5. **동작·품질**  
   삼분할 판정 로직, 시점 목록, 후보 수집, 경계 정밀화, 병합, 최소 길이 필터는 그대로이므로  
   **검출되는 구간의 집합은 기존과 동일**해야 합니다.  
   다만 ffmpeg의 디코딩 경로(한 번에 긴 구간 vs 여러 번 짧은 시크)에 따라 **같은 시점에서 약간 다른 프레임**이 나올 수는 있어,  
   이론적으로는 극히 예외적인 경우 판정이 달라질 여지는 있습니다.  
   실사용에서는 동일한 결과를 기대할 수 있습니다.
