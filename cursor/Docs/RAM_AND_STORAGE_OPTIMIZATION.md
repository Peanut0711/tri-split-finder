## RAM·NVMe 활용 정리 (Tri-Split Finder)

### 1. 목표

- **목표**: 보유한 CPU, GPU, RAM(DDR5 64GB), NVMe 4.0 SSD를 활용해  
삼분할 구간 검출·자르기·병합을 **최대한 빠르게** 수행하는 방법 정리.

### 2. 어디가 병목인지

- 병목은 보통 **연산(CPU/GPU)** 보다는 **I/O(네트워크/디스크)** 에서 발생.
- 특히:
  - **코스 스캔·경계 정밀화**: 원본 영상에서 프레임를 여러 번 읽어야 함.
  - **병합(--merge)**: 원본에서 여러 구간을 읽고, TS/MP4를 여러 번 쓰고 읽음.
- 원본이 **네트워크/NAS** 에 있을 때는, NVMe 속도보다 **네트워크 대역폭 + NAS HDD** 가 진짜 병목이 되는 경우가 많음.

### 3. RAM(DDR5 64GB) 활용 전략

#### 3.1 원본을 RAM에 올려서 처리 (가장 효과 큼)

- **아이디어**: 원본이 네트워크/NAS일 때, 실행 전에 **한 번만** RAM 디스크로 복사해 두고,
`tripartite_section_check.py` 의 입력을 그 RAM 디스크 경로로 주기.
- **효과**:
  - 코스 스캔, 경계 정밀화, 병합이 모두 **네트워크/HDD 읽기 → RAM 읽기** 로 바뀜.
  - I/O 병목이 크게 줄어듦.
- **용량**:
  - 3시간 FHD TS/MP4는 대략 20~40GB 수준인 경우가 많음.
  - DDR5 64GB면 한 번에 RAM 디스크에 올려 처리하는 시나리오가 충분히 가능.
- **실행 흐름 예시**:
  1. RAM 디스크 생성 (예: `R:\`).
  2. `\\NAS\video\foo.ts` → `R:\foo.ts` 로 복사.
  3. `python tripartite_section_check.py R:\foo.ts --cuda --workers 16 --segments-out ...`
  4. 필요시 병합도 동일한 RAM 경로로 입력을 주고 실행.

#### 3.2 병합 임시 폴더를 RAM 디스크에 두기

- 스크립트는 병합 시 **임시 폴더**로 `Videos/tripartite_merge_tmp/merge_xxx` 를 사용.
- 이 임시 경로를 **RAM 디스크 위**로 옮기면:
  - 구간 cut 결과 TS 쓰기,
  - concat 시 TS 읽기/쓰기,
  - 키프레임 트림 결과 쓰기
  전부 RAM에서 일어나서 병합 단계가 빨라짐.
- **주의**:
  - 구간/길이에 따라 임시 TS 총 용량이 수~십 GB까지 갈 수 있으므로 RAM 여유를 고려.
  - NVMe 4.0 SSD만으로도 충분히 빠른 경우가 많고, RAM 디스크는 “더 줄이고 싶을 때” 선택.

#### 3.3 코스 캐시를 RAM에 두기 (고급)

- `--coarse-cache` 를 디스크 대신 RAM(공유 메모리, RAM 디스크 등)에 올릴 수 있음.
- **의미가 있는 경우**:
  - 코스 스캔을 **원본 해상도**로 돌리고,
  - 경계 정밀화에서 “가까운 시각”이면 캐시 프레임을 재사용하는 구조일 때.
- **의미가 적은 경우**:
  - `--coarse-scale 640` 같은 **저해상도 코스**에서는 경계 정밀화가 항상 원본 해상도 프레임을 요구하므로,
  코스 캐시는 경계 단계에서 사용되지 않음.
- 구현 난이도에 비해 이득이 상대적으로 작으므로,
**1) 원본 RAM 디스크 + 2) 병합 임시 RAM 디스크**를 먼저 적용하는 것을 권장.

### 4. NVMe 4.0 SSD는 충분한가?

- **로컬 NVMe 4.0** 은 순차 읽기/쓰기 수 GB/s 수준으로,
  - 원본이 로컬에 있을 때,
  - 임시 TS/concat 결과/최종 MP4를 전부 NVMe에 둘 때,
  I/O 병목이 되기 어려움.
- 병목은 보통:
  - 원본/출력이 **네트워크 경로(NAS + 1Gbit LAN)** 에 있을 때,
  - 또는 HDD 기반 네트워크 스토리지의 **seek + 대역폭** 에서 발생.
- 따라서:
  - **원본/출력을 네트워크에 둘 수밖에 없다**면 → NVMe 4.0은 “로컬 작업용 버퍼”로는 이미 충분하고,  
  병목은 네트워크/NAS 쪽이므로 **RAM 디스크(또는 NVMe)에 원본을 옮겨서 처리**하는 전략이 중요.
  - **원본이 로컬 NVMe**라면 → 이 작업량에서는 NVMe 4.0만으로도 충분하며,
  RAM은 “더 줄이고 싶을 때” 병합 임시·원본 캐싱용으로 선택적으로 활용.

### 5. 추천 적용 순서

1. **검출·병합 모두 로컬 NVMe에서 실행**
  - 원본을 NVMe로 복사 가능하다면 가장 먼저 적용.
  - `--merge` 출력/임시도 NVMe 경로를 사용.
2. **원본을 RAM 디스크로 복사 후 실행**
  - 원본이 원래 네트워크/NAS에 있을 때,
  - 64GB 내에 들어오는 영상은 RAM 디스크에 올려서 전체 파이프라인을 RAM 기반으로 돌림.
3. **병합 임시 폴더를 RAM 디스크로**
  - NVMe 단계까지 최적화한 뒤, 병합 시간을 더 줄이고 싶을 때 적용.
4. **코스 캐시를 RAM에 두는 것은 선택**
  - 원본 해상도 코스 + 경계에서 캐시 사용 구조일 때만 효과.
  - 저해상도 코스(`--coarse-scale`) 위주의 세팅에서는 우선순위 낮음.

### 6. 네트워크 대용량(10~20GB+)·RAM 복사 FAQ

#### 6.1 10~20GB 넘는 파일도 RAM에 올려서 할 만한가?

- **예, 충분히 할 만함.**
- **용량**: 10~20GB × 1파일이면 DDR5 64GB에서는 전혀 부담되지 않음. OS·기타 프로그램·여유를 빼도 20GB급 1~2개는 여유 있게 들어감.
- **복사 시간 (대략)**  
  - 1Gbit LAN 기준 실효 대역폭 ~80~100MB/s 가정 시:
    - 10GB → 약 100~130초 (1분 40초~2분 10초)
    - 20GB → 약 200~260초 (3~4분)
  - 그 뒤 검출·병합은 **네트워크를 반복해서 읽지 않고 RAM 디스크만 읽으므로**, 코스 스캔·경계·병합 구간의 I/O 시간이 크게 줄어듦.

#### 6.2 RAM에 복사하는 게 부담(비용)이 크지 않나?

- **한 번만 옮기는 비용**이면, 위처럼 3~4분 수준.
- **여러 번 돌릴 영상**이면:  
  네트워크에서 매번 10~20GB를 읽는 것보다, **한 번 RAM으로 당겨 둔 뒤 N번 읽는 쪽이 훨씬 싸게 먹힘** → 부담이 커지는 게 아니라 **총 작업 시간이 줄어드는 쪽**에 가깝음.
- **딱 한 번만 돌리고** 이미 전체 실행이 5분 이내로 충분히 빠르다면, 굳이 RAM 복사까지 할 필요는 없음.  
  반대로 **한 번 실행에 15~20분** 걸린다면, 앞에서 3~4분 복사하고 뒤를 5~7분으로 줄이는 쪽이 훨씬 이득.

#### 6.3 “이중 일처리”가 아니냐?

- **아님.**  
  - **이중 일처리**처럼 느껴지는 건 “같은 일을 두 번 한다”는 의미인데,  
    여기서는 **데이터를 한 번 당겨 온 뒤, 그걸 여러 단계(코스·경계·병합)에서 반복해서 읽는 것**이라서,  
    네트워크를 여러 번 읽는 것보다 **한 번 복사 + RAM에서 N번 읽기**가 훨씬 효율적임.
- **한 번만 처리하는 영상**이어도:  
  복사 3~4분 + 작업 5분 vs 네트워크에서 그대로 10~12분 → 전체가 줄어드는 경우가 많음.
- **여러 번 처리하는 영상**이면:  
  RAM에 한 번 올려두고 그 뒤 N번 작업을 전부 RAM에서 하면, **이중이 아니라 “한 번 당겨서 N번 재사용”**이라 오히려 권장되는 패턴.

**정리**: 10~20GB 네트워크 영상을 RAM으로 복사해서 쓰는 건 64GB 기준으로 부담되지 않고, 이중 일처리가 아니라 **한 번 복사로 전체 I/O를 줄이는 전략**에 가깝다.
