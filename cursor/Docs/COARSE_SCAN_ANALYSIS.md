# 코스 스캔(Coarse Scan) 원리 및 소요 시간 분석

전체 실행 시간의 **71.5%**가 코스 스캔에 쓰일 때, 그 원리·사용 자원·왜 오래 걸리는지를 정리한 문서입니다.

---

## 1. 코스 스캔이란?

**목적**: 3~6시간 같은 장편 영상에서 “삼분할인 구간”을 빠르게 찾기 위해, **전체를 30초 간격으로 샘플링**해 각 시점이 삼분할 O/X인지 먼저 찍어 보는 단계입니다.

- 이후 **경계 정밀화**에서는 “삼분할이 시작/끝나는 정확한 초”만 이진 탐색으로 구합니다.
- 즉, **코스 스캔 = 굵게 훑기**, **경계 정밀화 = 구간 경계만 촘촘히 찾기**입니다.

---

## 2. 동작 원리 (코드 기준)

### 2.1 시점 목록 생성

- 영상 길이 `duration`(초)과 `--coarse-interval`(기본 30초)로 **샘플 시점 리스트**를 만듭니다.
- `t = 0, 30, 60, …` 으로 `t < duration` 까지 증가시키고, 끝이 빠지지 않도록 마지막에 `duration - 0.1` 정도를 한 번 더 넣습니다.

**예시 (길이 02:58:21 ≈ 10701초, 간격 30초)**  
→ 시점 개수 ≈ `10701 / 30 + 1` ≈ **358개**.

### 2.2 각 시점마다 하는 일 (`_coarse_worker`)

각 시점 `t`에 대해 **한 워커**가 다음 두 가지만 수행합니다.

1. **`extract_frame(path, t, width, height, use_gpu)`**  
   - **ffmpeg**를 **한 번** 실행해서, 시점 `t`에서 **프레임 1장**만 디코딩해 메모리로 가져옵니다.
   - `--cuda` 이면 `-hwaccel cuda` 로 GPU 디코딩 시도.
2. **`is_tripartite_frame(frame, left_right_only, use_gpu)`**  
   - 가져온 1프레임으로 삼분할 여부 판정 (높이 크롭 → 좌/중/우 분할 → 정렬 허용 시 L 탐색 → MSE 계산).

즉, **코스 스캔 = “시점 358개 × (ffmpeg 1회 + 삼분할 판정 1회)”** 입니다.

### 2.3 병렬화 (`--workers 16`)

- **`ProcessPoolExecutor(max_workers=16)`** 로 **프로세스 16개**를 띄웁니다.
- 시점 358개를 **청크**로 나눠 16개 프로세스에 분배하고, 각 프로세스는 자신이 받은 시점들에 대해 위 1→2를 **독립적으로** 반복합니다.
- 워커당 시점 수 ≈ 358 / 16 ≈ **22~23개** → 워커 하나가 **ffmpeg 22~23회 + 판정 22~23회** 수행.

---

## 3. 사용 자원

| 자원 | 사용 방식 |
|------|-----------|
| **CPU** | 워커 16개 = **Python 프로세스 16개**. 각 워커가 ffmpeg 자식 프로세스를 생성하므로, 동시에 **최대 16개의 ffmpeg**가 돌 수 있음. |
| **GPU** | `--cuda` 시 **ffmpeg**가 `-hwaccel cuda`로 디코딩. 워커 16개가 동시에 ffmpeg를 띄우면 **동시에 여러 ffmpeg가 CUDA 디코더 사용** 가능 → GPU 메모리·디코더 대기 등으로 **경합** 가능. CuPy로 삼분할 판정도 GPU 사용 시, 16프로세스가 같은 GPU를 쓰면 역시 **경합**. |
| **디스크 I/O** | 같은 영상 파일을 **여러 ffmpeg가 서로 다른 시점으로 seek**하며 읽음. 16프로세스가 동시에 읽으면 **랜덤 액세스 부하**가 커짐 (HDD면 특히 불리). |
| **메모리** | 프레임 1장 ≈ 1920×1080×3 ≈ **6MB**. 워커 16개 × (프레임 + 크롭/분할 버퍼) → 수백 MB 수준. |

정리하면, **코스 스캔은 “프로세스 16개 × (ffmpeg 1회 + 판정 1회)”를 시점 수만큼 반복**하는 구조이고, 그 중심에는 **시점당 1회씩 ffmpeg 실행**이 있습니다.

---

## 4. 왜 오래 걸리는가? (71.5%가 코스 스캔인 이유)

### 4.1 시점 수가 많다

- 30초 간격이면 **약 3시간에 360회**, **매 시점마다 ffmpeg 1회 + 판정 1회**.
- 시점 수를 줄이지 않는 한, 이 횟수만큼 **최소 비용**이 정해져 있음.

### 4.2 시점마다 ffmpeg를 새로 띄운다 (가장 큰 원인)

- **`extract_frame()`** 안에서 매번 **`subprocess.run(ffmpeg, ...)`** 로 **프로세스 한 번 새로 실행**합니다.
- ffmpeg는 매 호출마다:
  - 프로세스 생성
  - 영상 파일 열기
  - `-ss time_sec` 로 **시점 seek**
  - 해당 구간부터 **최소 1프레임 디코딩** (키프레임까지 뒤로 가서 디코딩할 수 있음)
  - raw 비디오를 stdout으로 출력
- TS 등에서는 **seek가 무거울 수 있고**, **프로세스 기동 + 파일 오픈**이 **시점 수만큼** 반복되므로, **시점 수 × (seek + 디코딩 + 프로세스 오버헤드)** 가 그대로 코스 스캔 시간의 주성분이 됩니다.

즉, **“시점 358개 = ffmpeg 358번 실행”** 이라서, 코스 스캔이 전체의 71.5%를 차지하는 것은 자연스러운 구조입니다.

### 4.3 병렬화 한계

- **workers 16** 이어도:
  - **358 / 16 ≈ 22** 이므로, **한 워커가 22번** ffmpeg를 순차 실행합니다.
  - ffmpeg 1회가 평균 0.5초만 걸려도 **워커당 11초**, 16개 워커가 동시에 돌아도 **이론상 11초대**이지만, 실제로는:
    - **디스크/네트워크 I/O**가 16개가 겹치면 지연
    - **GPU**를 16개 ffmpeg + 16개 CuPy가 나누어 쓰면 **디코더/연산 경합**으로 효율이 떨어질 수 있음
- 그래서 **워커를 늘려도** “시점당 ffmpeg 1회” 구조가 바뀌지 않으면, **선형으로만 빨라지고** 71.5% 비중이 크게 줄지는 않습니다.

### 4.4 정렬 허용(ALIGN_TOLERANCE_PX)의 영향

- 각 프레임에 대해 **L을 수십 번** 바꿔 가며 MSE를 계산합니다.
- 이 부분은 **ffmpeg보다는 가벼운 편**이지만, 시점 수가 많고 ffmpeg 비중이 압도적이라서, **전체 시간 대비 코스 스캔 71.5%**의 주된 이유는 여전히 **“시점 수 × ffmpeg 1회”** 구조입니다.
- 정렬 허용이 **어느 정도·어떻게** 검사되는지는 아래 **§7** 참고.

## 5. 요약 표

| 항목 | 내용 |
|------|------|
| **코스 스캔이 하는 일** | 30초 간격 시점 리스트(예: 358개)를 만들고, 각 시점에서 **ffmpeg로 1프레임 추출 → 삼분할 판정** |
| **쓰는 자원** | CPU(워커 16프로세스), GPU(ffmpeg CUDA + CuPy, 경합 가능), 디스크 I/O(동시 다중 seek) |
| **오래 걸리는 핵심 이유** | **시점마다 ffmpeg를 새 프로세스로 한 번씩 실행** (시점 수 × (seek + 디코딩 + 프로세스 오버헤드)) |
| **71.5%인 이유** | 시점 수가 많고, 그만큼 ffmpeg 호출이 반복되며, 경계 정밀화 등 다른 단계는 “구간 개수”에만 비례해 상대적으로 가벼움 |

## 6. 실행 옵션과의 관계 (참고)

- **`python tripartite_section_check.py "F:\...\input.ts" --cuda --workers 16`**  
  - **코스 스캔**: 30초 간격 시점들에 대해 **16개 프로세스**가 각각 **ffmpeg(CUDA) + 삼분할 판정** 수행.
- **코스 스캔 시간을 줄이고 싶다면**  
  - **`--coarse-interval 60`** 또는 **`120`** 으로 시점 수를 줄이기 (정밀도는 떨어짐),  
  - 또는 추후에 “ffmpeg 한 번 띄워서 여러 시점을 한 스트림으로 추출”하는 방식으로 구조를 바꾸는 개선이 필요합니다.

## 7. 정렬 허용(ALIGN_TOLERANCE_PX / --align-tolerance) 검사 방식

가상선이 640/1280이 아닌 647/1273처럼 **몇 픽셀 어긋난** 경우에도 삼분할로 인식하기 위한 옵션입니다. **CLI `--align-tolerance PX`**로 지정 가능하며, 미지정 시 스크립트 상단 상수 `ALIGN_TOLERANCE_PX`(기본 5)를 사용합니다.

### 7.1 오차 범위 (얼마나 허용하는가)

| 항목 | 값 | 설명 |
|------|-----|------|
| **상수/CLI** | `ALIGN_TOLERANCE_PX = 5` (기본) | `tripartite_section_check.py` 상단. **0**이면 정렬 탐색 없이 고정 1/3·2/3만 사용. CLI `--align-tolerance N`으로 변경 가능. |
| **기준 위치** | `base = w // 3` | 크롭 영역 가로 `w`에 대해 “이상적인 좌측 너비”. FHD 1920이면 **640** 픽셀. |
| **탐색 범위** | `L ∈ [base − N, base + N]` (N=5 기본) | **1픽셀 단위**로만 시도. 1920 기준·기본 5면 **635 ~ 645** (총 11개 정수). N=30이면 610~670. |
| **상한** | `L ≤ crop_w // 2` | 좌/우가 같은 픽셀 수가 되려면 `L`이 화면 절반 이하여야 함. 1920이면 960 이하. |

즉, **“좌측(및 우측) 너비 L”**을 **640±N 픽셀**(기본 N=5: 635~645) 구간에서 **1픽셀 간격**으로 바꿔 가며 검사합니다.

### 7.2 검사 방법 (어떻게 검사하는가)

- **분할**: 각 후보 `L`에 대해  
  - `left = region[:, :L]`  
  - `right = region[:, w - L :]`  
  (좌측 L픽셀, 우측 L픽셀 — 대칭으로 잡음.)
- **판정**  
  - **좌·우만 비교 모드**(기본): `MSE(left, right)` 한 번만 계산.  
  - **strict 모드**: `MSE(left, center)`, `MSE(right, center)` 각각 계산.
- **탐색 방식**  
  - **`is_tripartite_frame`** (코스 스캔·경계 정밀화에서 사용):  
    `L`을 **base−N ~ base+N** 순서로 시도하다가, **한 번이라도** `MSE ≤ MSE_THRESHOLD(500)` 이면 즉시 **True** 반환. (최적 L을 저장하지 않음.) 기본 N=5면 11개 후보.  
  - **`get_tripartite_mse`** (--verify·이미지 저장용):  
    위와 같은 **L 후보들**을 전부 시도하고, **MSE가 가장 작은 L**을 `best_L`로 저장한 뒤, 그때의 MSE로 OK/NG 판정 및 export 시 그 `best_L`로 좌/중/우 이미지 분할.

**서브픽셀·그래디언트 검색은 없음.** 정수 픽셀 단위만 사용합니다.

### 7.3 요약

| 질문 | 답 |
|------|----|
| 오차를 **얼마나** 허용하나? | **좌측 너비 L을 640±N 픽셀** 범위에서 탐색. 기본 N=5 (`ALIGN_TOLERANCE_PX = 5`). CLI `--align-tolerance N`으로 변경 가능. |
| **어떻게** 검사하나? | L을 **1픽셀 간격**으로 후보들을 넣어 보며, 각 L마다 `left`/`right`(·`center`)로 MSE 계산. 하나라도 500 이하면 OK; verify 시에는 MSE 최소인 L을 `best_L`로 씀. |

이 문서는 `tripartite_section_check.py` 코드(시점 생성, `_coarse_worker`, `extract_frame`, `ProcessPoolExecutor` 사용 부분, `ALIGN_TOLERANCE_PX`, `_is_tripartite_frame_*`, `get_tripartite_mse`)를 기준으로 정리한 내용입니다.
